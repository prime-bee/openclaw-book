Início/Capítulos/Capítulo 27: Migrando de um para múltiplos gateways

Migrar de um único gateway para múltiplos gateways não é um exercício de heroísmo. É um trabalho de observação, isolamento, ensaio e corte controlado de tráfego.

Capítulo 27 de 34AvançadoMigration · Rollback · Parallel run

Como migrar de um gateway para múltiplos gateways sem quebrar operação

Depois que um OpenClaw começa a crescer, surge uma tentação comum: separar tudo rapidamente em múltiplas instâncias. O problema é que a migração "virando a chave" costuma ser o caminho mais curto para indisponibilidade, colisão de credenciais e confusão operacional. A estratégia certa é rodar em paralelo, validar e só depois mover tráfego aos poucos.

Diagrama — Fases da migração

Antes de tudo: confirme que múltiplos gateways são necessários

Se o problema for apenas separar personas, workspaces ou rotas de mensagem, multi-agent no mesmo gateway ainda costuma ser a solução melhor. Vá para múltiplos gateways quando você precisa de isolamento mais forte de processo, config, portas, auth ou risco.

Plano detalhado

Fase 1 — Mapear o estado atual

  1. Liste canais ativos e contas vinculadas.
  2. Mapeie o workspace atual e os workspaces derivados.
  3. Mapeie o state dir atual.
  4. Registre a porta base, portas derivadas de browser/CDP e qualquer override manual.
  5. Defina qual função a nova instância vai assumir: rescue, lab, work, segregação por risco, etc.

Fase 2 — Subir a nova instância em paralelo

  1. Crie um profile novo.
  2. Escolha uma porta base bem separada.
  3. Garanta state/config/workspace exclusivos.
  4. Suba a instância nova sem desmontar a antiga.
  5. Valide `status`, `browser status` e logs da nova sem tocar no principal.

Fase 3 — Ensaiar o comportamento

  1. Teste canais e ferramentas na nova instância.
  2. Verifique se browser/CDP não colidem.
  3. Confira se as credenciais certas estão no profile certo.
  4. Faça um plano de rollback simples e escrito.

Fase 4 — Virar tráfego aos poucos

  1. Migre um canal ou uma função de cada vez.
  2. Observe estabilidade, latência e ergonomia operacional.
  3. Não migre tudo enquanto o comportamento ainda estiver "estranho mas funcionando".
  4. Depois de estabilizar, continue a expansão.

Estratégia de rollback

Rollback bom não é drama. É preparação. Você precisa conseguir voltar um canal, uma rota ou uma operação para a instância anterior sem inventar enquanto a casa já está pegando fogo.

Pergunta
Se você não sabe responder, ainda não está pronto
Qual canal vai primeiro?
defina um corte pequeno
Como volta?
tenha a reversão documentada antes
Quem guarda as credenciais?
não replique desorganização entre instâncias
Como saber se deu certo?
status, logs, health, browser status e observação humana

Anti-padrões de migração

  • copiar config e só trocar a porta
  • reaproveitar o mesmo state dir
  • subir duas instâncias e só depois lembrar do browser/CDP
  • migrar todos os canais de uma vez
  • não documentar qual instância é responsável por quê

Validação mínima

openclaw --profile main status
openclaw --profile rescue status
openclaw --profile rescue browser status
openclaw doctor
openclaw health

Em uma migração saudável, a instância nova demonstra saúde operacional antes de receber tráfego relevante.
