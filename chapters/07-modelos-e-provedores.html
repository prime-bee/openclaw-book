Modelos: quem responde

No OpenClaw, o modelo não é fixo para sempre. A documentação mostra uma hierarquia importante: modelo primário, fallbacks e, em alguns casos, modelo específico para imagem.

Ordem de seleção

  1. modelo primário
  2. fallbacks em ordem
  3. failover interno de auth do provedor
Intuição útil: o gateway decide qual modelo tentar; o provedor pode ainda trocar de perfil/credencial antes de desistir.

Comandos importantes

openclaw models status
openclaw models list
openclaw models set openai/gpt-5.2
openclaw models fallbacks add anthropic/claude-sonnet-4-5

Boas práticas

  • use um modelo forte como principal
  • configure fallbacks para robustez
  • se trabalha com imagens, pense no image model separadamente
  • evite allowlists excessivamente restritivas se você troca de modelo com frequência

Problema clássico: "model is not allowed"

Se existe uma allowlist em agents.defaults.models, o OpenClaw bloqueia modelos fora dela. Isso pode parecer que o agente "sumiu", mas na verdade ele recusou a troca.

Gestão de tokens e custos

Tokens são a unidade de cobrança de todos os provedores de LLM. Entender como eles funcionam é essencial para operar agentes de forma econômica e previsível.

Como tokens são contados

Cada chamada a um modelo envolve dois fluxos de tokens:

  • Tokens de entrada (input): tudo que entra no contexto — system prompt, histórico de conversa, conteúdo de ferramentas, arquivos injetados.
  • Tokens de saída (output): a resposta gerada pelo modelo. Geralmente cobrada a uma taxa mais alta (2–5× o input em muitos provedores).
  • Tokens de imagem: imagens são convertidas em tokens conforme resolução. Uma imagem de 1024×1024 pode custar 1.000–5.000 tokens dependendo do provedor e do detalhe solicitado.
  • Tokens de ferramentas: as definições de ferramentas (tool schemas) são enviadas em cada chamada e também consomem tokens de entrada. Um conjunto de 10 ferramentas bem detalhadas pode custar 2.000–5.000 tokens extras por chamada.

Estratégias para reduzir consumo de tokens

  1. Prompts concisos: cada palavra no system prompt é cobrada em toda chamada. Prefira instruções diretas e elimine redundâncias. Um system prompt de 800 tokens tem custo diferente de um de 200 tokens multiplicado por todas as sessões do mês.
  2. Truncação de histórico: configure um limite razoável de turnos de conversa no contexto. Não há valor em carregar 50 mensagens antigas para responder uma pergunta simples. Use maxHistoryTurns ou equivalente para limitar o contexto ativo.
  3. Context compaction: algumas plataformas e frameworks permitem comprimir o histórico em um resumo antes que ele encha o contexto. Implemente sumarização periódica para sessões longas.
  4. Seleção seletiva de ferramentas: não envie todas as ferramentas em toda chamada. Se a tarefa é uma resposta de chat simples, não há necessidade das ferramentas de browser ou de execução de código.
  5. Modelos menores para tarefas simples: classificação, extração de entidades simples e reformatação de texto podem ser feitos com modelos leves a uma fração do custo.

Cache de prompt (prompt caching)

A Anthropic oferece uma funcionalidade chamada prompt caching que permite reutilizar partes do contexto entre chamadas consecutivas. Quando habilitado, blocos marcados com cache_control são armazenados no servidor por até 5 minutos (ou mais em planos enterprise) e reutilizados sem reprocessamento completo.

{
  "system": [
    {
      "type": "text",
      "text": "Você é o assistente técnico da equipe de dados...",
      "cache_control": { "type": "ephemeral" }
    }
  ]
}

Quando usar:

  • System prompts longos usados em muitas sessões consecutivas (economia típica: 60–90% do custo do bloco cacheado)
  • Documentos de referência que são incluídos em múltiplas chamadas
  • Exemplos few-shot extensos que não mudam entre chamadas

Quando não usar:

  • Contextos que mudam a cada chamada (sem ganho)
  • Sessões únicas de baixo volume (overhead de setup não compensa)
Referência: veja a documentação completa de prompt caching em docs.anthropic.com/en/docs/build-with-claude/prompt-caching.

Critérios de seleção de modelos

A escolha do modelo é uma das decisões mais impactantes na operação de um agente. Não existe resposta universal — ela depende da natureza da tarefa, das exigências de latência, do orçamento e do nível de confiabilidade necessário.

O triângulo velocidade × qualidade × custo

Todo modelo vive nesse triângulo. Otimizar os três simultaneamente é impossível — você sempre troca algum pilar pelos outros:

  • Modelos grandes (ex: Claude Opus, GPT-4o): máxima qualidade de raciocínio, melhor em tarefas ambíguas e complexas, mais lentos, mais caros. Use quando a qualidade da decisão é crítica.
  • Modelos intermediários (ex: Claude Sonnet, GPT-4o-mini): equilíbrio prático. Bons para a maioria das tarefas operacionais de agentes. Velocidade razoável, custo moderado.
  • Modelos leves (ex: Claude Haiku, Gemini Flash): muito rápidos, muito baratos. Excelentes para classificação, triagem, extração simples e tarefas onde erros são recuperáveis.

Estratégia de fallback em cascata

Uma abordagem robusta é tentar primeiro o modelo mais rápido e barato, escalando para modelos mais capazes apenas quando necessário. Isso é chamado de cascata de modelos:

Etapa 1: Haiku responde (barato, rápido)
  → se confiança baixa ou tarefa complexa detectada:
Etapa 2: Sonnet responde (intermediário)
  → se ainda inadequado ou erro crítico:
Etapa 3: Opus responde (melhor qualidade)

No OpenClaw, isso pode ser implementado via:

  1. Configurar fallbacks no gateway com modelos ordenados por capacidade
  2. Usar roteamento condicional no agente coordenador (ver capítulo 8)
  3. Avaliar a qualidade da resposta do modelo leve antes de usar

Quotas e limites de taxa

Provedores de LLM impõem limites para proteger a infraestrutura e garantir acesso equitativo. Entender esses limites é fundamental para agentes de produção que fazem muitas chamadas ou lidam com picos de uso.

Tipos de limites

  • RPM (Requests Per Minute): número máximo de chamadas à API por minuto. Afeta sistemas de alta frequência como bots de chat com muitos usuários ou loops de agentes rápidos.
  • TPM (Tokens Per Minute): total de tokens processados por minuto. Mais restritivo para sessões com contextos grandes ou muitos documentos.
  • TPD (Tokens Per Day): alguns planos têm limite diário absoluto. Útil monitorar para evitar cortes inesperados em produção.
  • Concurrent requests: alguns provedores limitam chamadas simultâneas, afetando arquiteturas paralelas de agentes.

Impacto nos agentes

Quando um agente bate no limite de taxa, ele recebe um erro 429 Too Many Requests. Sem tratamento adequado, isso causa:

  • Falha silenciosa da tarefa agendada (cron jobs que não concluem)
  • Interrupção de loops de raciocínio do agente no meio da execução
  • Cascata de erros em sistemas multi-agent onde agentes dependem uns dos outros

Estratégias de fila e backoff

  1. Exponential backoff: ao receber 429, espere antes de tentar novamente. Dobre o intervalo a cada tentativa (ex: 1s, 2s, 4s, 8s) com jitter aleatório para evitar thundering herd.
  2. Fila com prioridade: em sistemas multi-tenant, enfileire requisições e processe por prioridade. Tarefas críticas passam na frente de rotinas de baixa prioridade.
  3. Rate limiting proativo: implemente um token bucket ou leaky bucket no próprio gateway para nunca enviar mais do que 80% do limite contratado, deixando margem para picos.
  4. Distribuição entre provedores: use fallbacks para provedores alternativos quando o provedor primário estiver saturado. O sistema de fallback do OpenClaw é exatamente para isso.
# Exemplo de lógica de backoff (pseudocódigo)
async function chamadaComRetry(payload, maxTentativas = 5) {
  let delay = 1000; // 1 segundo inicial
  for (let i = 0; i < maxTentativas; i++) {
    try {
      return await chamarModelo(payload);
    } catch (err) {
      if (err.status !== 429) throw err; // outro erro, não retry
      if (i === maxTentativas - 1) throw err; // esgotou tentativas
      const jitter = Math.random() * 500;
      await sleep(delay + jitter);
      delay *= 2; // backoff exponencial
    }
  }
}

Monitoramento de consumo de quota

Monitore ativamente para evitar surpresas:

  • Registre tokens consumidos por chamada nos logs do gateway (input, output, total)
  • Configure alertas quando o consumo diário atingir 70% do limite contratado
  • Use openclaw logs --follow para inspecionar chamadas e identificar sessões que consomem mais do esperado
  • Agrupe consumo por agente e por tipo de tarefa para identificar onde está o gasto
  • Revise periodicamente se modelos maiores estão sendo usados onde modelos menores seriam suficientes
Dica operacional: um spike inesperado de tokens geralmente aponta para um loop de agente sem condição de parada, um arquivo muito grande sendo injetado no contexto em toda chamada, ou um cron job mal configurado que dispara com frequência excessiva. Cheque essas causas primeiro.

Exercícios

Exercício 1: Por que faz sentido ter fallbacks em um sistema que depende de APIs externas? Pense em dois cenários concretos onde um fallback evita indisponibilidade.
Exercício 2: Mapeie as tarefas do seu agente atual e classifique cada uma em "leve", "intermediária" ou "pesada" em termos de complexidade de raciocínio. Para quais você poderia substituir o modelo atual por um mais barato sem perda perceptível de qualidade?
Exercício 3: Calcule o custo mensal estimado do seu heartbeat. Se ele roda a cada 30 minutos, com contexto de 1.000 tokens de entrada e 200 de saída, quantos tokens são consumidos por dia? Por mês? Qual seria a economia ao usar um modelo 10× mais barato para essa tarefa?

Checklist antes de trocar provedor

  1. confirme se o modelo desejado está permitido na allowlist
  2. verifique credenciais, limites e região do provedor
  3. teste um prompt curto antes de migrar fluxos críticos
  4. deixe fallback explícito para evitar quedas silenciosas
  5. valide os limites de RPM/TPM do novo provedor contra o volume esperado
  6. revise o custo estimado por tarefa no novo provedor antes da migração
