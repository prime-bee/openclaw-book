Controle de navegador

O OpenClaw também pode operar navegadores, inspecionar tabs, clicar, digitar e navegar. Isso abre espaço para automação visual e workflows de web.

Perfis

  • openclaw — navegador dedicado
  • user — sua sessão já existente
  • chrome-relay — aba controlada por extensão

Comandos básicos

openclaw browser profiles
openclaw browser start
openclaw browser open https://example.com
openclaw browser snapshot
openclaw browser screenshot

Ações

  • navegar
  • clicar
  • digitar
  • tirar screenshot
  • inspecionar snapshot

Controle remoto

Se o browser está em outra máquina, a doc recomenda um node host nessa máquina. O gateway então faz proxy dos comandos.

Isso importa: browser automation útil em produção quase sempre exige pensar em sessão, login, segurança e onde o navegador realmente roda.

Padrões de automação robusta

Browser automation em produção falha por razões previsíveis: elementos que ainda não carregaram, seletores que mudaram com uma atualização do site, redes lentas, timeouts insuficientes. Automação robusta não tenta evitar falhas — ela as antecipa e lida com elas graciosamente.

Estratégias de espera

O erro mais comum em automação de browser é clicar ou ler antes que o elemento esteja pronto. Existem três estratégias de espera, em ordem crescente de robustez:

Estratégia
Como funciona
Quando usar
Espera por tempo fixo
Aguarda N segundos sem verificar
Nunca em produção. Frágil e lento por definição.
Espera por seletor
Aguarda até que um elemento específico apareça no DOM
Para elementos com seletor estável. Mais rápido que tempo fixo.
Espera por estado
Aguarda até que o elemento esteja visível, clicável ou tenha um valor específico
Preferido. Verifica estado real, não só presença no DOM.
Princípio: espere pelo estado que você precisa, não pelo tempo que você acha que será suficiente. Páginas lentas fazem timeout; páginas rápidas desperdiçam espera fixa.

Lógica de retry e tratamento de erros

Em automação robusta, toda ação crítica deve ter uma política de retry explícita. Erros transitórios — elemento temporariamente fora do DOM, rede lenta, elemento coberto por outro — são normais em produção.

Padrão de retry para automação OpenClaw

# No system prompt do agente de automação:
Se uma ação de browser falhar, siga este protocolo:
1. Aguarde 2 segundos
2. Tire um screenshot para diagnóstico
3. Verifique se a página mudou (novo URL, novo título)
4. Tente a ação novamente
5. Se falhar pela segunda vez, registre o erro com screenshot
   e escalate para o operador humano

Não repita mais de 3 vezes a mesma ação sem sucesso.
Registre cada tentativa no log de sessão.

Checklist de tratamento de erros em automação

  • Capture screenshot em falhas: um screenshot no momento da falha vale mais do que dez linhas de log. Instrua o agente a sempre tirar screenshot quando uma ação falhar.
  • Registre o estado da página: URL atual, título da página e presença de elementos chave ajudam a diagnosticar o que deu errado.
  • Defina limites de retry: sem limite, um loop de retry pode consumir cota de API indefinidamente. Três tentativas é o máximo razoável para a maioria dos casos.
  • Escalate quando necessário: se após as tentativas a ação ainda falha, o agente deve parar e notificar o operador — não improvisar.
  • Diferencie erros transitórios de permanentes: elemento não encontrado pode ser transitório; página de erro 404 é permanente. Tratamentos diferentes.

CDP: capacidades do Chrome DevTools Protocol

O CDP (Chrome DevTools Protocol) é o protocolo de baixo nível que o OpenClaw usa para controlar o navegador. Ele expõe capacidades além do clique e navegação básica:

Interceptação de rede

Via CDP, é possível interceptar e inspecionar requisições de rede. Útil para capturar respostas de API que a página consome, monitorar chamadas de autenticação, ou verificar que um recurso foi carregado.

Caso de uso: aguardar que uma chamada de API específica complete antes de continuar a automação.

Emulação de dispositivo

CDP permite emular viewport de mobile, densidade de pixel e touch events. Para automações que precisam testar comportamento mobile ou sites que servem conteúdo diferente por device.

Caso de uso: capturar screenshot da versão mobile de uma página.

Execução de JavaScript

CDP permite executar JavaScript diretamente no contexto da página. Útil para extrair dados de variáveis JS, disparar eventos customizados, ou interagir com elementos que resistem a clique normal.

Caso de uso: extrair dados do estado interno de uma aplicação React ou Vue.

Captura de performance

CDP expõe métricas de performance da página: tempo de carregamento, consumo de memória, layout shifts. Útil para monitoramento de performance automatizado.

Caso de uso: alerta automático quando o tempo de carregamento de uma página crítica aumenta acima de um threshold.

CDP e segurança: execução de JavaScript via CDP tem acesso ao DOM completo, cookies e localStorage da página. Use com cuidado em páginas autenticadas — um script mal escrito pode vazar dados de sessão ou corrompê-los.

Boas práticas de estabilidade

Automações frágeis criam mais trabalho do que economizam. Estas práticas reduzem a taxa de falha em produção:

Seletores resistentes a mudança

Tipo de seletor
Estabilidade
Recomendação
ID único (#checkout-button)
Alta
Prefira quando disponível. IDs raramente mudam entre deploys.
Atributo de dados ([data-testid="submit"])
Alta
Excelente. Atributos data-* são mantidos para automação.
Texto do elemento (text=Confirmar)
Média
Bom para elementos com texto estável. Frágil se o texto muda com i18n.
Classes CSS (.btn-primary)
Baixa
Evite. Classes mudam com redesigns e atualizações de framework.
XPath absoluto
Muito baixa
Nunca use. Quebra com qualquer mudança de estrutura HTML.

Timeouts e configuração defensiva

Configuração de timeout para automação em produção

# Instrução no system prompt do agente de browser:
Ao automatizar páginas web, use estes limites:
- Timeout para carregar página: 30 segundos
- Timeout para aguardar elemento: 10 segundos
- Timeout para ação (clique, digitação): 5 segundos
- Máximo de retentativas por ação: 3

Se qualquer timeout for excedido:
1. Capture screenshot imediatamente
2. Registre o estado atual da página (URL, título)
3. Notifique o operador com o contexto completo
4. Não tente continuar o fluxo — pare e escalate

Receita de scraping com resiliência

Scraping é um dos usos mais comuns de browser automation. Uma receita anti-frágil funciona mesmo quando a página está lenta, parcialmente carregada ou com elementos faltando.

Receita: scraping resiliente passo a passo

  1. Defina o dado mínimo: antes de começar, identifique quais campos são obrigatórios e quais são opcionais. A receita deve ter sucesso parcial — não falhar tudo se um campo opcional não estiver presente.
  2. Navegue e aguarde o estado certo: não leia imediatamente após navegar. Aguarde que o elemento principal da página esteja visível e estável.
  3. Capture snapshot antes de extrair: um snapshot do DOM no momento da extração serve como evidência e facilita debug posterior.
  4. Extraia com fallback: para cada campo, defina um seletor primário e um seletor alternativo. Se o primário falhar, tente o alternativo antes de registrar como "não encontrado".
  5. Valide o que foi extraído: antes de salvar, verifique se o dado faz sentido — por exemplo, um preço deve ser numérico, uma data deve ser parseável.
  6. Salve com metadados: junto com os dados extraídos, salve: URL de origem, timestamp, versão do seletor, e se houve fallback. Facilita auditar a qualidade dos dados depois.
  7. Registre falhas explicitamente: se um campo obrigatório não foi encontrado, registre o erro com contexto e não apenas "null". Isso acelera o diagnóstico quando o site muda.
Anti-fragilidade: uma automação anti-frágil não apenas tolera falhas — ela coleta informação sobre as falhas que encontra. Com o tempo, o log de falhas revela os padrões de instabilidade da fonte, permitindo melhorar os seletores e a lógica de retry progressivamente.

Exercício

Projeto: imagine uma automação segura usando browser. O que poderia ser automatizado sem expor demais sua conta?

Pré-voo para automação de browser

  • confirme se o Chrome/Chromium está acessível no host certo
  • defina se o caso exige perfil persistente ou sessão descartável
  • teste navegação e captura antes de automatizar cliques complexos
  • registre limites do ambiente remoto: latência, auth, telas e viewport
