Um gateway, vários cérebros

A doc de multi-agent deixa uma coisa clara: um agente não é só um nome. Ele tem workspace próprio, agentDir próprio, sessões próprias e, idealmente, credenciais próprias.

O que define um agente

  • workspace
  • state/agentDir
  • store de sessões
  • auth profiles
  • persona e regras

Por que isso é poderoso?

  • um agente para vida pessoal
  • outro para trabalho
  • outro para automações
  • outro para grupos específicos

Roteamento

Bindings determinam qual mensagem vai para qual agente. A especificidade importa: peer exato tende a ganhar de regras mais genéricas de canal.

peer específico > conta/canal específico > fallback do agente padrão

Exemplo mental

Você pode ter um agente "chat" atendendo WhatsApp e um agente "opus" atendendo Telegram. Ou um único canal com exceções por pessoa/grupo.

Regra de ouro: não compartilhe o mesmo agentDir entre agentes. Isso mistura auth e sessões.

Comando útil

openclaw agents add work
openclaw agents list --bindings

Padrões de arquitetura multi-agent

Quando múltiplos agentes colaboram em uma tarefa, a forma como eles se organizam define a confiabilidade, a escalabilidade e a facilidade de depuração do sistema. Existem quatro padrões fundamentais.

Hub-and-spoke: coordenador com especialistas

Um agente central (coordenador) recebe o pedido, decompõe em subtarefas e delega para agentes especializados. Os resultados retornam ao coordenador para síntese final.

         [Usuário]
              |
        [Coordenador]
       /      |       \
[Pesquisa] [Código] [Revisão]

Quando usar:

  • Tarefas compostas que exigem especialidades distintas
  • Quando a orquestração lógica é complexa e centralizar facilita o controle
  • Quando você precisa de um ponto único de auditoria e log

Pipeline: saída de um alimenta o próximo

Cada agente processa e passa o resultado para o próximo, formando uma cadeia linear. Ideal quando há etapas sequenciais com dependências claras.

[Coletor] → [Limpeza] → [Análise] → [Formatador] → [Entregador]

Quando usar:

  • ETL de dados: coletar, transformar, carregar
  • Fluxos de publicação: rascunho → revisão → formatação → publicação
  • Qualquer fluxo onde a ordem é estrita e cada etapa tem responsabilidade única

Votação e consenso: múltiplos agentes, melhor resposta

Vários agentes (geralmente idênticos ou com personas ligeiramente diferentes) produzem respostas independentes para o mesmo problema. Um árbitro (que pode ser outro agente ou lógica determinística) seleciona ou sintetiza a melhor resposta.

Pergunta → [Agente A] ┐
           [Agente B] ├→ [Árbitro] → Melhor resposta
           [Agente C] ┘

Quando usar:

  • Decisões críticas onde a confiança numa única resposta é insuficiente
  • Avaliação de risco com múltiplos critérios
  • Geração criativa onde diversidade é desejável antes da seleção final

Execução paralela: fan-out para subtarefas independentes

A tarefa é decomposta em subtarefas sem dependência entre si. Todas são disparadas simultaneamente e os resultados são reunidos quando todas concluem.

Tarefa grande
    ├→ [Agente 1: subtarefa A] ─┐
    ├→ [Agente 2: subtarefa B] ─┤ → Reunião de resultados
    └→ [Agente 3: subtarefa C] ─┘

Quando usar:

  • Processamento de múltiplos documentos simultaneamente
  • Consultar diferentes fontes de dados em paralelo
  • Tarefas de longa duração onde paralelismo reduz o tempo total

O papel do agente coordenador

O coordenador é o cérebro do sistema multi-agent. Sua qualidade determina a qualidade de todo o pipeline. Um bom coordenador precisa ter responsabilidades bem definidas e um system prompt que as reflita.

Responsabilidades do coordenador

  • Decomposição de tarefas: receber um pedido complexo e dividir em subtarefas claras, com escopo bem definido para cada especialista.
  • Delegação: enviar cada subtarefa ao agente correto, incluindo contexto suficiente mas não excessivo.
  • Síntese de resultados: reunir as respostas parciais e construir uma resposta coerente e útil para o usuário final.
  • Tratamento de falhas: detectar quando um subagente falhou ou retornou resultado inadequado, e decidir entre retry, fallback ou escalada para humano.

System prompt padrão para coordenador

Você é o coordenador de um sistema multi-agent.
Sua função é:
1. Receber tarefas do usuário
2. Decompor em subtarefas específicas
3. Delegar cada subtarefa ao agente especializado correto
4. Reunir os resultados e sintetizar uma resposta final

Agentes disponíveis:
- agente-pesquisa: busca e coleta de informações externas
- agente-codigo: análise, geração e revisão de código
- agente-redacao: criação e revisão de texto

Regras de operação:
- Sempre confirme que entendeu a tarefa antes de decompor
- Se uma subtarefa falhar 2 vezes, informe o usuário e peça orientação
- Nunca invente resultados; use apenas o que os subagentes retornaram
- Registre no workspace cada delegação e resultado para auditoria

Detectando e tratando falhas de subagentes

Falhas em sistemas multi-agent são inevitáveis. O coordenador precisa:

  1. Detectar timeouts: se o subagente não respondeu em um tempo razoável, não fique esperando indefinidamente. Defina um timeout explícito.
  2. Avaliar qualidade da resposta: uma resposta que chegou pode ainda ser inadequada (muito vaga, claramente incorreta, fora do escopo solicitado). O coordenador deve checar antes de usar.
  3. Estratégia de retry: falhas transitórias (timeout, rate limit) merecem retry com backoff. Falhas estruturais (subagente mal configurado) devem ser escaladas.
  4. Escalada para humano: quando o sistema não consegue resolver autonomamente, o coordenador deve apresentar o estado parcial ao usuário e pedir orientação. Veja padrão HITL no capítulo 16.

Isolamento e comunicação entre agentes

Agentes em produção precisam de fronteiras claras. Sem isolamento, contextos se contaminam, credenciais vazam e a depuração torna-se impossível.

Padrões de comunicação agent-to-agent

  • Workspace compartilhado com namespacing: agentes leem e escrevem em arquivos de workspace, mas cada um tem sua própria pasta. O coordenador escreve a tarefa em workspace/tarefas/pesquisa.md, o agente de pesquisa lê e escreve o resultado em workspace/resultados/pesquisa.md.
  • Passagem de mensagens via canal: o coordenador envia a tarefa como mensagem direta ao subagente pelo canal configurado. Mais simples, mas cria dependência de entrega de mensagem.
  • Fila de tarefas em workspace: crie um arquivo workspace/fila.json onde o coordenador enfileira tarefas e os agentes fazem polling. Robusto para volumes maiores.

Prevenindo contaminação de contexto

Contaminação acontece quando o contexto de uma tarefa influencia indevidamente outra. Para evitar:

  • Use sessões isoladas (--session isolated) para cada subtarefa independente
  • Nunca passe o histórico completo de conversa do coordenador para o subagente — passe apenas o contexto relevante para a subtarefa
  • Limpe o workspace de resultados parciais após a síntese final
  • Prefira IDs de tarefa únicos para que resultados não se misturem

Níveis de confiança entre agentes

Uma pergunta fundamental em sistemas multi-agent: o agente A deve confiar cegamente no resultado do agente B?

A resposta é: não automaticamente. O output de um subagente é, do ponto de vista do coordenador, um dado externo que precisa ser validado antes de uso. Considere:

  • Validação de formato: o subagente deveria retornar JSON? Verifique antes de parsear.
  • Validação de conteúdo: o resultado está dentro do escopo esperado? Um agente de pesquisa não deveria estar executando código.
  • Validação de completude: o resultado endereça de fato a subtarefa delegada? Resultados vagos ou genéricos devem ser rejeitados.
  • Prompt injection via subagente: se um subagente processa conteúdo externo (páginas web, e-mails), seu resultado pode conter instruções maliciosas. Trate o resultado como dado não-confiável (veja capítulo 10).

Receita: sistema de 3 agentes (coordenador + 2 especialistas)

Sistema mínimo: coordenador + pesquisa + redação

  1. Crie os agentes: 
    openclaw agents add coordenador
openclaw agents add pesquisa
openclaw agents add redacao
  2. Configure workspaces separados em config.yaml para cada agente, garantindo que cada um tenha seu próprio diretório de estado e memória.
  3. Configure o system prompt do coordenador (use o template da seção anterior) no arquivo de persona do agente coordenador.
  4. Configure o agente de pesquisa com acesso a ferramentas de busca e browser, e system prompt focado em: buscar, verificar fontes, retornar síntese factual com referências.
  5. Configure o agente de redação com acesso ao workspace compartilhado e system prompt focado em: receber briefing, consultar resultados de pesquisa, redigir conteúdo no formato especificado.
  6. Defina o protocolo de comunicação: crie um arquivo workspace/compartilhado/README.md documentando as pastas e convenções de nomes que cada agente deve usar.
  7. Teste o fluxo: envie uma tarefa ao coordenador pelo Telegram (ex: "Pesquise e escreva um resumo sobre prompt caching da Anthropic"). Monitore com openclaw logs --follow.
  8. Valide o isolamento: confirme com openclaw agents list --bindings que cada agente responde apenas aos canais e peers corretos.
# Estrutura de workspace compartilhado
workspace/
  compartilhado/
    tarefas/          # Coordenador escreve aqui
    resultados/       # Subagentes escrevem aqui
    sintetizado/      # Coordenador escreve resultado final
  coordenador/        # Estado privado do coordenador
  pesquisa/           # Estado privado do agente de pesquisa
  redacao/            # Estado privado do agente de redação

Anti-padrões de roteamento e multi-agent

Anti-padrões a evitar

  • criar múltiplos agentes sem diferença real de função, workspace ou credencial
  • misturar ambientes pessoais e operacionais no mesmo agente "coringa"
  • adicionar bindings demais sem documentar a precedência
  • achar que multi-agent resolve má organização editorial ou má configuração base
  • passar o contexto completo da conversa do coordenador para subagentes — isso polui o contexto e infla o custo
  • confiar cegamente no output de subagentes sem validação mínima de formato e conteúdo
  • usar sessões compartilhadas entre agentes — use sempre sessões isoladas para subtarefas independentes

Exercício

Desenhe seu setup ideal: se você tivesse 3 agentes, quais seriam? Um para quê, outro para quê, e como você os separaria? Qual padrão de arquitetura (hub-and-spoke, pipeline, votação, paralelo) melhor se aplica ao seu caso de uso?
Exercício avançado: implemente uma validação simples de output no coordenador. Defina 3 critérios que uma resposta de subagente deve atender antes de ser aceita. O que acontece quando ela falha nesses critérios?
