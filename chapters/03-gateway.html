Gateway: o coração do OpenClaw

Segundo a runbook oficial, o Gateway é o processo sempre ligado que faz roteamento, RPC, HTTP, UI e conexões de canais.

Comandos essenciais

openclaw gateway --port 18789
openclaw gateway status
openclaw gateway status --deep
openclaw status
openclaw logs --follow
Atalho mental: pense no gateway como o ponto onde “transporte, controle e observabilidade” se encontram. Se ele falha, quase todo o resto parece quebrado.

O que ele concentra

  • WebSocket de controle
  • APIs HTTP
  • UI de controle
  • canais conectados
  • estado de sessões
  • saúde operacional

Porta e bind

A doc destaca uma precedência clara para porta e bind. Em geral, a porta padrão é 18789 e o bind padrão é loopback, o que reduz exposição acidental.

Regra prática: expor o gateway para fora exige mais cuidado com autenticação. A doc enfatiza token/senha, Tailscale ou túnel SSH.

Operação do serviço

Você pode rodar em foreground durante testes ou instalar como serviço para ficar sempre disponível.

openclaw gateway install
openclaw gateway start
openclaw gateway restart
openclaw gateway stop

Checklist de operação saudável

  1. Confirme que o processo subiu e responde em openclaw gateway status.
  2. Verifique se o bind e a porta estão coerentes com o nível de exposição desejado.
  3. Cheque logs logo após restart para capturar erros precoces de auth, config ou porta.
  4. Se houver canais/nós, valide-os antes de declarar o ambiente “ok”.

Quando usar foreground vs serviço

Cenário
Escolha recomendada
Teste rápido, ajuste de config, leitura de logs em tempo real
Foreground, para observar comportamento imediatamente
Uso contínuo, reinício automático, operação diária
Serviço instalado, com monitoramento e rotina de restart consciente
Diagnóstico após incidente
Comece pelo status/logs; só reinicie depois de formular hipótese

Verificações úteis

  • openclaw gateway status --deep
  • openclaw channels status --probe
  • openclaw health

Falhas comuns

  • EADDRINUSE: porta em uso
  • unauthorized: token/senha incorretos
  • bind inseguro: tentativa de bind fora de loopback sem auth adequada

Sequência curta de diagnóstico

openclaw gateway status --deep
openclaw logs --follow
openclaw channels status --probe
openclaw health

Se o gateway sobe mas o comportamento está estranho, tente isolar a camada do problema: primeiro runtime/bind/auth, depois canais e por fim políticas de roteamento.
