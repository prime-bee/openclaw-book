Receitas práticas: do zero ao funcionando

Este capítulo transforma a documentação em checklists executáveis. A ideia é reduzir atrito: você copia a sequência mental, adapta os detalhes e valida o resultado.

Receita 1 — Subir o Gateway e validar saúde

  1. Instale/configure o OpenClaw com openclaw onboard se ainda não fez o bootstrap.
  2. Inicie o serviço com openclaw gateway --port 18789 ou use o serviço instalado.
  3. Valide com openclaw gateway status e openclaw status.
  4. Se algo parecer estranho, acompanhe openclaw logs --follow.
openclaw gateway --port 18789
openclaw gateway status
openclaw status
openclaw logs --follow
Sinal de sucesso: runtime running + RPC probe ok.

Receita 2 — Colocar Telegram para funcionar

  1. Crie um bot no BotFather.
  2. Adicione o token em config.
  3. Defina uma política mínima de acesso.
  4. Suba o gateway.
  5. Aprove o primeiro pairing se usar pairing.
{
  channels: {
    telegram: {
      enabled: true,
      botToken: "123:abc",
      dmPolicy: "pairing",
      groups: { "*": { requireMention: true } }
    }
  }
}
Erro comum: esperar que openclaw channels login telegram exista. Telegram usa token em config/env, não login via CLI.

Receita 3 — Criar um heartbeat útil sem desperdiçar tokens

  1. Mantenha o HEARTBEAT.md curto.
  2. Defina intervalo razoável (30m ou 1h).
  3. Escolha target: none se não quiser notificações, last se quiser.
  4. Teste observando se ele retorna HEARTBEAT_OK quando nada importa.
{
  agents: {
    defaults: {
      heartbeat: {
        every: "30m",
        target: "last",
        lightContext: true
      }
    }
  }
}

Receita 4 — Criar um cron job de briefing matinal

openclaw cron add \
  --name "Morning brief" \
  --cron "0 7 * * *" \
  --tz "America/New_York" \
  --session isolated \
  --message "Faça um resumo do dia" \
  --announce \
  --channel telegram \
  --to "123456789"

Depois valide com:

openclaw cron list
openclaw cron runs --id <jobId> --limit 20

Receita 5 — Criar um segundo agente

  1. Crie o agente: openclaw agents add work.
  2. Defina workspace separado.
  3. Crie bindings por canal, conta ou peer.
  4. Verifique com openclaw agents list --bindings.
Dica: multi-agent faz mais sentido quando há diferença real de persona, credenciais, workspace ou canal.

Receita 6 — Publicar no GitHub Pages com revisão mínima

  1. rode a validação local do site
  2. confirme links principais e metadados
  3. faça merge em master
  4. acompanhe a action de deploy e abra o Pages publicado
python3 scripts/validate-site.py
node --check app.js
node --check search.js
git push origin master
Sinal de sucesso: o deploy conclui e a navegação principal funciona sem regressões visíveis.

Receita 7 — Monitoramento de site com alerta automático

Configure um agente para monitorar periodicamente um site, detectar mudanças ou erros, e enviar alerta no Telegram.

  1. Configure um cron job de monitoramento que roda a cada hora (ou conforme necessário): 
    openclaw cron add \
  --name "Monitor site principal" \
  --cron "0 * * * *" \
  --tz "America/Sao_Paulo" \
  --session isolated \
  --message "Acesse https://meusite.com e verifique: (1) a página carrega sem erro; (2) o título é o esperado; (3) o elemento .header existe. Se encontrar qualquer problema, reporte com: ALERTA: [descrição do problema]. Se tudo estiver ok, responda apenas: OK" \
  --channel telegram \
  --to "SEU_CHAT_ID"
  2. Configure a ferramenta de browser no agente para permitir snapshots de URL sem interação (modo leve para economizar tokens).
  3. Defina o critério de alerta no system prompt do agente: quais mudanças são relevantes (ex: status HTTP diferente de 200, texto específico ausente, tempo de carga acima de 5s).
  4. Implemente lógica de silêncio: para não receber "OK" a cada hora, configure o target como none no cron e mude para last apenas quando detectar problemas. Ou instrua o agente a só enviar mensagem quando detectar anomalias.
  5. Adicione contexto de baseline no workspace: crie workspace/monitoramento/baseline.md com o estado esperado do site (título, elementos-chave, estrutura) para que o agente compare com o atual.
  6. Valide o fluxo com um teste manual: simule um erro (ex: aponte para uma URL inválida) e confirme que o alerta chega no Telegram.
Dica: use openclaw cron runs --id <jobId> --limit 20 para revisar o histórico de execuções e detectar padrões de falha.

Receita 8 — Processamento de inbox de e-mail com triagem

Configure um agente para ler resumos de e-mail, classificar por urgência e tópico, criar itens de ação no workspace e rotear para o canal correto.

  1. Prepare a fonte de e-mails: o agente pode receber resumos de e-mail via mensagem direta (você copia e cola) ou via integração com uma ferramenta que lê o inbox (IMAP, Gmail API, etc.). Para o caso mais simples, crie um cron que você alimenta manualmente.
  2. Crie o template de triagem no system prompt do agente: 
    Para cada e-mail recebido, classifique segundo:
URGÊNCIA: [crítico|alto|médio|baixo]
TÓPICO: [financeiro|técnico|comercial|pessoal|spam|outro]
AÇÃO: [responder|delegar|arquivar|ignorar]
PRAZO: [hoje|esta semana|sem prazo]
RESUMO: [1 linha]
  3. Configure ação por classificação:
    • Crítico + responder → notificação imediata no Telegram com resumo
    • Alto + esta semana → item de ação em workspace/inbox/acoes.md
    • Médio/baixo → log em workspace/inbox/triagem.md sem notificação
    • Spam → log e ignorar
  4. Implemente o roteamento por canal: e-mails técnicos vão para o agente de tecnologia (se houver multi-agent), comerciais para o canal do time de vendas, pessoais ficam apenas no workspace privado.
  5. Crie um resumo semanal de inbox: adicione um cron na sexta às 17h que lê workspace/inbox/triagem.md e produz um sumário das ações pendentes e e-mails processados na semana.
  6. Valide o fluxo: envie 5 e-mails simulados (um de cada categoria) e confirme que a triagem e o roteamento funcionam como esperado.
Privacidade: nunca armazene o conteúdo completo de e-mails sensíveis no workspace. Guarde apenas classificação, metadados e resumos anonimizados conforme necessário.

Receita 9 — Geração de relatório semanal automatizado

Um cron na sexta à noite dispara um agente que lê as entradas da semana no workspace e produz um relatório estruturado entregue no Discord ou Telegram.

  1. Configure o cron de relatório semanal: 
    openclaw cron add \
  --name "Relatório semanal" \
  --cron "0 18 * * 5" \
  --tz "America/Sao_Paulo" \
  --session isolated \
  --message "Leia todas as entradas desta semana em workspace/diario/ e workspace/tarefas/. Produza um relatório semanal estruturado com: (1) Realizações da semana; (2) Problemas encontrados e como foram resolvidos; (3) Tarefas abertas com prioridade; (4) Aprendizados e observações. Formato: markdown, tom profissional." \
  --channel discord \
  --to "SEU_CANAL_DISCORD"
  2. Mantenha um diário operacional: durante a semana, registre eventos relevantes em workspace/diario/YYYY-MM-DD.md. O agente usará esses arquivos como fonte do relatório.
  3. Configure o formato de saída: instrua o agente no system prompt a usar headers markdown, listas e, quando relevante, métricas numéricas (tarefas concluídas, incidentes, etc.).
  4. Adicione contexto de metas: crie workspace/metas-semana.md com as metas semanais definidas toda segunda. O agente incluirá na análise se as metas foram atingidas.
  5. Configure entrega multi-canal: envie o relatório completo no Discord e um resumo executivo (3 pontos) no Telegram pessoal.
  6. Valide em semana de teste: crie entradas artificiais no diário e acione o cron manualmente para confirmar que o formato e conteúdo estão adequados antes de usar em produção.
Variação útil: adapte para relatório diário às 23h, resumindo o dia, ou relatório mensal no último dia útil do mês para uma visão mais ampla de progresso.

Receita 10 — Pipeline de revisão de código com agente e HITL

Quando notificado de um PR, o agente lê o diff, verifica problemas comuns e prepara um rascunho de revisão que só é publicado após aprovação humana.

  1. Configure a notificação de PR: use um webhook do GitHub/GitLab que envia uma mensagem para o agente via Telegram ou canal configurado, contendo o número do PR e o diff resumido.
  2. Configure o agente revisor com system prompt especializado: 
    Você é um revisor de código assistente. Quando receber um diff:
1. Analise em busca de: (a) vulnerabilidades de segurança óbvias;
   (b) violações de estilo do projeto; (c) lógica potencialmente incorreta;
   (d) código morto ou redundante
2. Para cada problema encontrado, descreva: localização, severidade (crítico/alto/médio/baixo), explicação e sugestão de correção
3. Conclua com: APROVADO (sem problemas críticos) ou REQUER REVISÃO (problemas críticos encontrados)
4. Responda APENAS com o rascunho de revisão — NÃO publique nada ainda
  3. Implemente o checkpoint HITL: após o agente produzir o rascunho, envie para o humano responsável via Telegram com as opções: "Publicar como está", "Editar e publicar", ou "Descartar". O agente aguarda a resposta antes de qualquer ação.
  4. Configure a publicação: se aprovado, o agente usa a ferramenta de browser ou API do GitHub/GitLab para postar o comentário de revisão no PR.
  5. Registre o histórico de revisões em workspace/revisoes/PR-NUMERO.md com o diff recebido, o rascunho gerado, a decisão humana e o timestamp.
  6. Defina escopo claro de o que o agente verifica: foque em padrões objetivos (nomes de variáveis, imports não usados, tratamento de erro ausente) e não em opiniões subjetivas sobre arquitetura, para reduzir falsos positivos.
HITL essencial: nunca deixe o agente publicar comentários de revisão em PRs sem aprovação humana. Comentários incorretos ou inadequados em repositórios públicos têm consequências reais para a equipe e para a reputação do projeto.
