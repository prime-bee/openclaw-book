Skills: ensinar o agente a trabalhar melhor sem acumular dívida operacional

No OpenClaw, uma skill não é só “um comando extra”. Ela é uma camada de especialização: um pacote de instruções que ajuda o agente a decidir quando usar uma ferramenta, como usar, que cuidado tomar e qual formato de saída entregar. Em vez de aumentar a inteligência do modelo magicamente, a skill aumenta a qualidade operacional do sistema.

Modelo mental: ferramentas são as mãos; modelos são o cérebro; skills são os playbooks que ensinam esse cérebro a usar as mãos direito.

O que uma skill resolve na prática

  • Especialização: o agente aprende um fluxo recorrente sem precisar redescobrir tudo do zero.
  • Redução de erro: a skill codifica cuidados, limites e anti-padrões explícitos.
  • Escolha de ferramenta: deixa mais claro quando usar browser, shell, tmux, cron, memória, web ou uma API externa.
  • Menos ambiguidade: o agente sabe o que significa “fazer direito” naquele contexto.
  • Compartilhamento de conhecimento: você escreve uma vez e reutiliza em várias sessões ou agentes.

Onde as skills vivem

Segundo a documentação oficial, o OpenClaw carrega skills de três lugares:

Origem
Quando usar
Bundled skills
Vêm com a instalação do OpenClaw. Úteis como base comum e referência de implementação.
~/.openclaw/skills
Skills locais compartilhadas entre agentes da mesma máquina.
<workspace>/skills
Skills específicas de um projeto ou agente, com precedência mais alta.

Isso importa porque a skill certa, no lugar certo, evita tanto duplicação quanto vazamento de comportamento entre agentes que deveriam ser separados.

Precedência e isolamento

Se existir uma skill com o mesmo nome em mais de um lugar, a precedência é:

<workspace>/skills  →  ~/.openclaw/skills  →  bundled skills

Na prática:

  • se você está escrevendo comportamento específico deste projeto, prefira a skill no workspace;
  • se é algo reutilizável entre vários agentes seus, prefira a skill local gerenciada;
  • se quer manter compatibilidade ampla, estude primeiro como a bundled skill já faz.

Como pensar uma boa skill

Uma boa skill costuma responder cinco perguntas:

  1. Quando ela deve ser usada?
  2. Qual problema ela resolve?
  3. Quais ferramentas ela ensina a usar?
  4. Quais riscos ela precisa conter?
  5. Que resultado “bom” ela deve produzir?

Skill não é o mesmo que ferramenta

Ferramenta

Executa uma capacidade direta: ler arquivo, editar, rodar comando, abrir browser, buscar na web.

Skill

Explica como e quando usar uma ou várias ferramentas para resolver uma classe de problema.

Agente

Decide e coordena. A skill ajuda o agente a não improvisar mal em cenários recorrentes.

Exemplo prático:

  • Ferramenta: browser
  • Skill: “como usar browser automation com segurança e checkpoints humanos”
  • Resultado: o agente deixa de clicar aleatoriamente e passa a seguir um procedimento mais previsível.

Lifecycle de uma skill: da dor recorrente ao playbook estável

O erro clássico é escrever uma skill no impulso, usá-la duas vezes e esquecer que ela existe. Com o tempo você acumula um cemitério de playbooks sem dono, sem revisão e sem saber se ainda valem. Para evitar isso, trate cada skill como um pequeno produto operacional com lifecycle explícito.

dor recorrente
    ↓
protótipo local
    ↓
uso repetido em casos reais
    ↓
revisão de riscos e permissões
    ↓
versionamento + distribuição controlada
    ↓
observabilidade e manutenção
    ↓
deprecação ou promoção para skill compartilhada

Esse ciclo força duas perguntas úteis:

  • a skill ainda resolve uma dor real?
  • o custo de mantê-la ainda é menor que o benefício operacional?

Estágios práticos

Estágio
O que acontece
Rascunho
Fluxo ainda instável, uso local, instruções curtas, sem prometer demais.
Beta operacional
Casos reais validam o fluxo; começam a surgir troubleshooting, exemplos e limites explícitos.
Stable
Escopo congelado o suficiente para versionar, revisar permissões e distribuir com confiança.
Legacy / deprecated
A skill foi substituída, mudou de escopo ou acumula risco/dívida demais para continuar ativa.

Formato e gating

Skills no OpenClaw seguem o formato AgentSkills-compatible e usam um SKILL.md com frontmatter. Além disso, o sistema pode filtrar skills por:

  • OS suportado
  • presença de binários
  • variáveis de ambiente
  • configuração habilitada
  • watchers e overrides de config

Isso evita que o agente veja como “disponível” uma skill que depende de algo inexistente na máquina ou no sandbox.

Implicação importante: uma skill pode ser válida no host e inválida dentro da sandbox. Se ela depende de um binário, esse binário também precisa existir no ambiente isolado.

Versionamento: a disciplina mínima para não se perder

Skills mudam de comportamento. Às vezes você melhora a saída; às vezes muda permissões; às vezes troca uma ferramenta por outra. Sem versionamento, o time só descobre a mudança quando a operação já está estranha.

Mesmo em ambiente pequeno, vale adotar um modelo simples inspirado em semver:

Tipo de mudança
Como versionar
Correção editorial, clareza, troubleshooting
Patch
Novo caso de uso compatível, novo exemplo, novo guardrail
Minor
Mudança de escopo, ferramenta, premissa de segurança ou saída esperada
Major

Não precisa inventar uma esteira pesada. Mas registre pelo menos:

  • versão atual;
  • data da última revisão;
  • dono ou mantenedor;
  • dependências críticas;
  • breaking changes conhecidas.
Heurística útil: se uma mudança altera o que a skill faz ou o nível de poder que ela concede, trate como evento de governança, não como simples edição de texto.

Revisão de permissões: o ponto onde produtividade vira risco

Uma skill costuma encapsular uma combinação de ferramentas, e a combinação importa mais do que cada peça isolada. read + web_search é bem diferente de exec + acesso à rede + escrita em produção. Por isso, revisão de permissões precisa olhar para blast radius composto.

Matriz mínima de revisão

Nível
Exemplo
Revisão exigida
Baixo
Leitura, síntese, navegação local, validação sem efeitos externos
Checagem editorial e clareza de escopo
Médio
Edição de arquivos, geração de artefatos, execução local reversível
Revisão de rollback, idempotência e limites do workspace
Alto
Push, deploy, escrita externa, automação autenticada, shell com efeitos persistentes
Aprovação humana, critérios de auditoria e evidência de validação

Perguntas que toda revisão deveria responder

  1. A skill deixa claro quando pedir confirmação humana?
  2. Ela distingue bem ações internas de ações que saem da máquina?
  3. O fluxo é idempotente ou pelo menos reversível?
  4. Há risco de a skill mascarar um comando destrutivo como “rotina normal”?
  5. O texto ensina o agente a registrar validação, logs ou evidências?

Distribuição segura: local, compartilhada ou pública?

Nem toda skill boa deve virar skill pública. A decisão de distribuição deveria considerar acoplamento, segredos, premissas ambientais e custo de suporte.

Somente workspace

Use quando a skill depende de arquivos, convenções, hosts, workflows ou linguagem muito específicos do projeto.

Compartilhada localmente

Use quando a capability se repete em vários agentes seus, mas ainda depende da sua infraestrutura ou dos seus padrões operacionais.

Distribuição pública

Use apenas quando dependências, riscos, setup e limites estiverem suficientemente claros para terceiros.

Checklist antes de publicar em ClawHub ou compartilhar com o time

  • removeu nomes privados de hosts, canais, paths sensíveis e credenciais?
  • documentou binários, permissões e variáveis exigidas?
  • deixou claro o que a skill não deve fazer?
  • incluiu um caminho de rollback ou de recuperação quando algo falhar?
  • testou em ambiente menos privilegiado antes de promover?

Skills locais vs skills do projeto

Uma regra simples ajuda bastante:

Tipo
Coloque onde?
Aprendizado pessoal reutilizável
~/.openclaw/skills
Fluxo específico deste projeto
<workspace>/skills
Base da instalação
Bundled skills

Isso reduz o risco de um comportamento que faz sentido aqui contaminar outros agentes ou outros projetos sem querer.

Quando vale criar uma skill nova

Crie uma skill quando você perceber um padrão repetido com:

  • mais de uma ferramenta envolvida;
  • passos recorrentes;
  • riscos conhecidos;
  • saída desejada relativamente estável;
  • benefício claro de não reexplicar tudo a cada conversa.

Exemplos bons:

  • publicar um site estático com validação e checklists;
  • review editorial com critérios consistentes;
  • triagem de incidentes de gateway;
  • uso disciplinado de browser automation em tarefas autenticadas.

Quando NÃO vale criar skill

  • Quando é só um passo simples.
  • Quando o fluxo ainda muda toda hora.
  • Quando você não entendeu bem o problema ainda.
  • Quando o texto da skill ficaria mais confuso que um bom prompt direto.
Exercício: pense em uma tarefa que você já repetiu três vezes com agentes. Ela exigiu várias ferramentas? Teve erros recorrentes? Houve sempre o mesmo formato de saída? Se sim, provavelmente está pedindo uma skill.

Composição de capacidades sem virar espaguete operacional

O uso mais interessante de skills não é “colecionar comandos”. É montar capacidades compostas. O perigo é cair no outro extremo: juntar tantas skills, exceções e condições que o agente vira uma colcha de retalhos impossível de prever.

Padrão saudável de composição

skill de diagnóstico
    +
skill de execução controlada
    +
skill de validação
    +
skill de comunicação / handoff

Esse arranjo funciona porque cada skill tem papel claro. Já o padrão ruim é este:

skill-mestra-que-faz-tudo
    +
subregras escondidas
    +
dependências implícitas
    +
aprovações mal definidas

Critérios para compor sem criar dívida

  • Uma skill, uma responsabilidade predominante. Se ela diagnostica, executa, valida e publica ao mesmo tempo, provavelmente está grande demais.
  • Dependências explícitas. Se uma skill pressupõe outra, documente a relação e o motivo.
  • Handoffs legíveis. A saída de uma skill deve ser útil para a próxima, sem exigir adivinhação.
  • Permissões proporcionais. Não eleve o poder da composição inteira por causa de um caso raro.
  • Critério de remoção. Toda skill deve poder ser aposentada sem colapsar o sistema inteiro.
Sinal de dívida operacional: quando ninguém sabe mais qual skill disparou qual comportamento, ou quando a mesma tarefa produz saídas muito diferentes dependendo de combinações ocultas, a composição já passou do ponto.

Exemplo real de design

Imagine um agente editorial para manter documentação:

Capability
Skill ideal
Descobrir o que mudou
Skill de leitura de changelog, git log e páginas relevantes
Aplicar atualização
Skill de edição com critérios editoriais e limites de escopo
Validar site
Skill de validação local com checklist técnico
Publicar
Skill separada para branch, merge, push e evidência de validação

Note que “publicar” não deveria ficar misturado com “descobrir” ou “editar”. Separar bem esses momentos reduz erro e facilita auditoria.

Governança de mudanças: o que revisar antes de promover uma skill

  1. Escopo: a skill ainda faz só o que promete?
  2. Dependências: binários, APIs, credenciais e ambientes continuam válidos?
  3. Segurança: o texto criou atalhos perigosos, normalizou bypasses ou diluiu aprovação humana?
  4. Compatibilidade: agentes ou workspaces antigos vão quebrar com a mudança?
  5. Observabilidade: existe evidência suficiente para entender falhas e regressões?

Esse checklist conecta diretamente o capítulo 21 ao capítulo 20: mexer em skill é mexer em governança. E conecta também ao capítulo 29: sempre que uma mudança de skill causar regressão real, ela merece virar caso de eval.

Troubleshooting de lifecycle e distribuição

A skill “sumiu”

Cheque precedência, nome do diretório, gating por binário/OS e se a skill foi sobrescrita por versão mais local.

A skill aparece, mas não funciona

Verifique se as dependências existem no sandbox/host correto e se o fluxo assumiu paths ou credenciais que não estão presentes.

O agente ficou agressivo demais

Revise linguagem da skill, thresholds de aprovação e qualquer instrução que trate ações sensíveis como rotina inofensiva.

Distribuição virou caos

Padronize versionamento, dono, changelog mínimo e política de promoção entre rascunho, beta e stable.

FAQ rápido

Preciso versionar até skills pequenas?

Nem sempre formalmente, mas pelo menos registre data, dono e mudanças relevantes. O custo é baixo e evita confusão depois.

Posso colocar tudo numa skill só?

Pode, mas quase nunca deve. A skill monolítica parece prática no começo e vira ponto único de ambiguidade depois.

Quando uma skill merece ir para ClawHub?

Quando o valor dela sobrevive fora do seu ambiente e o texto explica setup, limites, riscos e dependências sem conhecimento implícito do autor.

Como saber se uma skill está velha?

Se depende de ferramenta que mudou, se a operação contorna a skill com prompts ad hoc ou se ninguém mais confia no comportamento dela, já passou da hora de revisar.
