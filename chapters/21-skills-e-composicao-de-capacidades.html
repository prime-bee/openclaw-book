Skills: ensinar o agente a trabalhar melhor

No OpenClaw, uma skill não é só “um comando extra”. Ela é uma camada de especialização: um pacote de instruções que ajuda o agente a decidir quando usar uma ferramenta, como usar, que cuidado tomar e qual formato de saída entregar. Em vez de aumentar a inteligência do modelo magicamente, a skill aumenta a qualidade operacional do sistema.

Modelo mental: ferramentas são as mãos; modelos são o cérebro; skills são os playbooks que ensinam esse cérebro a usar as mãos direito.

O que uma skill resolve na prática

  • Especialização: o agente aprende um fluxo recorrente sem precisar redescobrir tudo do zero.
  • Redução de erro: a skill codifica cuidados e anti-padrões explícitos.
  • Escolha de ferramenta: deixa mais claro quando usar browser, bash, node, tmux, API externa etc.
  • Menos ambiguidade: o agente sabe o que significa “fazer direito” naquele contexto.
  • Compartilhamento de conhecimento: você escreve uma vez e reutiliza em várias sessões ou agentes.

Onde as skills vivem

Segundo a documentação oficial, o OpenClaw carrega skills de três lugares:

Origem
Quando usar
Bundled skills
Vêm com a instalação do OpenClaw. Úteis como base comum e referência.
~/.openclaw/skills
Skills locais compartilhadas entre agentes da mesma máquina.
<workspace>/skills
Skills específicas de um projeto ou agente, com precedência mais alta.

Isso importa porque a skill certa, no lugar certo, evita tanto duplicação quanto vazamento de comportamento entre agentes que deveriam ser separados.

Precedência e isolamento

Se existir uma skill com o mesmo nome em mais de um lugar, a precedência é:

<workspace>/skills  →  ~/.openclaw/skills  →  bundled skills

Na prática:

  • se você está escrevendo comportamento específico deste livro/projeto, prefira a skill no workspace;
  • se é algo reutilizável entre vários agentes seus, prefira a skill local gerenciada;
  • se quer manter compatibilidade ampla, estude primeiro como a bundled skill já faz.

Como pensar uma boa skill

Uma boa skill costuma responder cinco perguntas:

  1. Quando ela deve ser usada?
  2. Qual problema ela resolve?
  3. Quais ferramentas ela ensina a usar?
  4. Quais riscos ela precisa conter?
  5. Que resultado “bom” ela deve produzir?

Skill não é o mesmo que ferramenta

Ferramenta

Executa uma capacidade direta: ler arquivo, editar, rodar comando, abrir browser, buscar na web.

Skill

Explica como e quando usar uma ou várias ferramentas para resolver uma classe de problema.

Agente

Decide e coordena. A skill ajuda o agente a não improvisar mal em cenários recorrentes.

Exemplo prático:

  • Ferramenta: `browser`
  • Skill: “como usar browser automation com segurança e checkpoints humanos”
  • Resultado: o agente deixa de clicar aleatoriamente e passa a seguir um procedimento mais previsível.

Formato e gating

Skills no OpenClaw seguem o formato AgentSkills-compatible e usam um `SKILL.md` com frontmatter. Além disso, o sistema pode filtrar skills por:

  • OS suportado
  • presença de binários
  • variáveis de ambiente
  • configuração habilitada
  • watchers e overrides de config

Isso evita que o agente veja como “disponível” uma skill que depende de algo inexistente na máquina ou no sandbox.

Implicação importante: uma skill pode ser válida no host e inválida dentro da sandbox. Se ela depende de um binário, esse binário também precisa existir no ambiente isolado.

Segurança: skills são poder, então exigem cuidado

Skills são ótimas para produtividade, mas também ampliam a superfície de risco. A documentação oficial é bem clara: skills de terceiros devem ser tratadas como código não confiável.

Riscos principais

  • Excesso de autonomia: a skill pode ensinar um fluxo agressivo demais.
  • Vazamento de segredos: env/config mal planejados podem expor chaves.
  • Aumento de blast radius: uma skill ruim pode guiar o agente para usar ferramentas poderosas sem checkpoints suficientes.
  • Dependência quebrada: skill exige binário, endpoint ou credencial ausente.

Boas práticas

  • Leia a skill antes de ativar.
  • Use sandbox quando o fluxo for arriscado.
  • Evite embutir segredos em prompts ou documentação.
  • Documente claramente quando pedir confirmação humana.
  • Prefira skills pequenas e focadas a skills “faz tudo”.

Skills locais vs skills do projeto

Uma regra simples ajuda bastante:

Tipo
Coloque onde?
Aprendizado pessoal reutilizável
~/.openclaw/skills
Fluxo específico deste projeto
<workspace>/skills
Base da instalação
Bundled skills

Isso reduz o risco de um comportamento que faz sentido aqui contaminar outros agentes ou outros projetos sem querer.

Quando vale criar uma skill nova

Crie uma skill quando você perceber um padrão repetido com:

  • mais de uma ferramenta envolvida;
  • passos recorrentes;
  • riscos conhecidos;
  • saída desejada relativamente estável;
  • benefício claro de não reexplicar tudo a cada conversa.

Exemplos bons:

  • publicar um site estático com validação e checklists;
  • review editorial com critérios consistentes;
  • triagem de incidentes de gateway;
  • uso disciplinado de browser automation em tarefas autenticadas.

Quando NÃO vale criar skill

  • Quando é só um passo simples.
  • Quando o fluxo ainda muda toda hora.
  • Quando você não entendeu bem o problema ainda.
  • Quando o texto da skill ficaria mais confuso que um bom prompt direto.
Exercício: pense em uma tarefa que você já repetiu três vezes com agentes. Ela exigiu várias ferramentas? Teve erros recorrentes? Houve sempre o mesmo formato de saída? Se sim, provavelmente está pedindo uma skill.

ClawHub e distribuição

Se a skill ficar boa o bastante e não tiver acoplamento demais ao seu ambiente, você pode pensar em distribuí-la via ClawHub. Antes disso, vale revisar:

  • se não depende de nomes privados de hosts, canais ou arquivos;
  • se descreve dependências de forma explícita;
  • se os riscos estão bem documentados;
  • se o comportamento é claro para outras pessoas.

Skills como composição de capacidades

O uso mais interessante de skills não é “colecionar comandos”. É montar capacidades compostas:

  • uma skill para diagnosticar
  • outra para automatizar
  • outra para publicar
  • outra para revisar

Com o tempo, o agente deixa de parecer um executor improvisado e passa a se comportar como um operador com playbooks. É aí que o sistema amadurece.
