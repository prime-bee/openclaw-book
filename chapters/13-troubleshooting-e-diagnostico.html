Diagnóstico: quando algo não funciona

Este capítulo resume a mentalidade de troubleshooting da doc oficial: primeiro confirme a saúde global, depois vá para a camada específica.

Escada universal de diagnóstico

openclaw status
openclaw gateway status
openclaw logs --follow
openclaw doctor
openclaw channels status --probe

Essa sequência cobre: runtime, RPC, logs, drift/config e saúde dos canais.

Princípio operacional

Regra de ouro: tente sair de "não funciona" para uma frase observável. Exemplo: "gateway sobe, mas o canal X não entrega em grupos" é muito melhor que "está quebrado".

Sintoma → hipótese → ação

Sintoma
O que checar primeiro
Sem respostas
Pairing pendente, mention gating, allowlist, logs de bloqueio
Gateway não sobe
Porta em uso, bind sem auth, config inválida, modo remoto/local errado
Cron não dispara
openclaw cron status, jobs listados, histórico de runs, cron habilitado
Heartbeat não fala
target, active hours, HEARTBEAT_OK, directPolicy, heartbeat skip reasons
Browser falha
Profile correto, processo do browser, CDP alcançável, Playwright/relay

Quatro playbooks curtos

1) "Nada responde"

  1. Rode openclaw status.
  2. Confirme o canal com openclaw channels status --probe.
  3. Veja se há pairing pendente.
  4. Leia os logs procurando mention required, pairing request ou blocked.

2) "O serviço existe, mas está parado"

openclaw gateway status --deep
openclaw logs --follow
openclaw doctor

Se aparecer EADDRINUSE, é conflito de porta. Se aparecer recusa de bind sem auth, falta token/senha para bind não-loopback.

3) "Canal conectado, mas mensagens não fluem"

O erro costuma ser de política e não de conexão. Revise:

  • dmPolicy / groupPolicy
  • allowFrom / groupAllowFrom
  • requireMention
  • permissões do bot/plataforma

4) "Browser não faz nada"

openclaw browser status
openclaw browser profiles
openclaw browser start --browser-profile openclaw
openclaw logs --follow

No perfil chrome-relay, lembre que a aba precisa estar realmente anexada pela extensão.

Ritmo de investigação

  1. Confirme o sintoma e a camada afetada.
  2. Colete evidências curtas: status, probe, logs recentes.
  3. Faça uma hipótese por vez em vez de mudar tudo ao mesmo tempo.
  4. Após corrigir, rode a mesma checagem que reproduzia o problema.

Ordem de escalonamento

  1. estado geral do runtime
  2. gateway e conectividade
  3. canal ou integração específica
  4. modelo, credencial ou política
  5. bug reproduzível com contexto suficiente para issue

Escalonar cedo demais sem evidência custa tempo; escalar tarde demais sem registrar sinais custa memória.

Diagrama de diagnóstico em árvore

Problema percebido
├─ Gateway está de pé?
│  ├─ Não → status / logs / doctor / port / auth
│  └─ Sim
├─ Canal está saudável?
│  ├─ Não → status --probe / token / pairing / scopes
│  └─ Sim
├─ Política está deixando passar?
│  ├─ Não → allowlist / mention / group policy
│  └─ Sim
└─ Ferramenta específica falhou?
   ├─ Browser → profile / cdp / playwright
   ├─ Cron → status / runs / delivery
   └─ Node → approvals / permissions / connectivity

Kit mínimo para incidentes

  • Um comando de status geral
  • Um comando da camada específica
  • Logs recentes do serviço
  • Uma hipótese explícita antes de reiniciar ou reconfigurar

Diagnóstico de qualidade de resposta

Problemas de qualidade são mais difíceis de diagnosticar do que problemas técnicos — não há um erro explícito, apenas respostas que decepcionam. Use este guia para identificar a causa raiz.

Agente dando respostas genéricas ou inúteis

Sintoma: o agente responde com frases vagas ("claro, posso ajudar com isso!"), não resolve a tarefa concreta, ou ignora o contexto específico.

Causas comuns e correções:

  • System prompt ausente ou vago: sem instrução de papel e escopo, o modelo entra em modo genérico. Corrija adicionando identidade, escopo e formato esperado ao system prompt (veja capítulo 18).
  • Contexto insuficiente na mensagem: o usuário enviou uma pergunta sem contexto. Instrua o agente a perguntar pelo contexto antes de responder quando a tarefa for ambígua.
  • Modelo inadequado para a tarefa: modelos leves (Haiku/Flash) têm capacidade limitada para raciocínio complexo. Considere escalar para um modelo mais capaz (veja capítulo 7).
  • Temperatura muito alta: configurações de temperatura elevada aumentam criatividade mas reduzem precisão. Para tarefas operacionais, use temperatura 0 ou próxima de 0.

Agente alucinando fatos

Sintoma: o agente afirma fatos incorretos com confiança, inventa referências, ou descreve funcionalidades que não existem.

Prevenção e correção:

  • Grounding com fontes: forneça documentos de referência no contexto ou via workspace. Instrua o agente a citar apenas o que está na fonte fornecida.
  • Instrução explícita de incerteza: adicione ao system prompt: "Se não tiver certeza, diga 'não tenho certeza' antes de responder. Nunca invente informações."
  • Verificação com ferramentas: configure ferramentas de busca ou acesso a workspace para que o agente consulte fontes antes de responder, em vez de depender apenas da memória do modelo.
  • Auditoria de respostas críticas: para domínios onde alucinação tem alto custo (médico, jurídico, financeiro), implemente um passo de verificação humana antes de qualquer ação baseada na resposta.

Agente ignorando instruções do system prompt

Sintoma: o agente não segue regras definidas no system prompt — responde em idioma errado, usa formato diferente do especificado, ou executa ações proibidas.

Diagnóstico e correção:

  1. Confirme que o system prompt está sendo enviado corretamente: verifique nos logs se a chamada inclui o campo system com o conteúdo esperado.
  2. Verifique conflito com o histórico de conversa: instruções anteriores no histórico podem sobrepor o system prompt. Limpe o histórico e teste novamente.
  3. Reformule instruções negativas como positivas: "não use bullets" funciona menos que "use parágrafos corridos".
  4. Priorize instruções críticas no início do system prompt, não no final — modelos tendem a dar mais peso ao início do contexto.
  5. Teste com uma sessão totalmente nova e contexto mínimo para isolar se o problema é do prompt ou do estado acumulado.

Agente em loop ou preso

Sintoma: o agente continua tentando a mesma ação sem progresso, ou fica chamando ferramentas repetidamente sem concluir a tarefa.

Detecção:

  • Monitore o número de tool calls por sessão nos logs — mais de 15-20 chamadas sem conclusão é sinal de loop
  • Configure um limite máximo de iterações no agente (max_steps ou equivalente)
  • Observe padrões repetitivos nos logs: mesma ferramenta chamada com os mesmos argumentos é diagnóstico claro

Interrupção:

  • Encerre a sessão corrente: openclaw sessions kill <sessionId>
  • Adicione ao system prompt uma condição explícita de parada: "Se tentar a mesma ação mais de 3 vezes sem progresso, pare e explique ao usuário o que está bloqueando."
  • Implemente um mecanismo de circuit breaker: se N tentativas falharem, escale para humano em vez de continuar tentando

Diagnóstico de performance

Performance degradada afeta diretamente a experiência do usuário e o custo operacional. Diagnostique antes de otimizar.

Alta latência de resposta

Sintoma: o agente leva muito tempo para responder — segundos a dezenas de segundos mesmo para perguntas simples.

Causas e diagnóstico:

  • Modelo muito grande para a tarefa: Claude Opus ou GPT-4o têm latência maior que modelos leves. Se a tarefa é simples, troque por Sonnet ou Haiku (veja a matriz do capítulo 7).
  • Contexto excessivo: contextos com 50.000+ tokens demoram mais para processar. Reduza o histórico de conversa e filtre arquivos desnecessários do workspace.
  • Muitas tool calls em sequência: cada tool call é uma ida e volta ao modelo. Redesenhe o fluxo para minimizar roundtrips — agrupe ações quando possível.
  • Latência de rede: use openclaw logs --follow e compare o tempo de início da chamada com o fim para separar latência de rede da latência de processamento do modelo.

Alto consumo de tokens

Sintoma: fatura acima do esperado ou rate limits sendo atingidos mais cedo que o previsto.

Diagnóstico com logging:

# Inspecione o consumo de tokens por sessão
openclaw logs --follow | grep "tokens"

# Identifique sessões com consumo anômalo
openclaw sessions list --sort tokens-desc --limit 10

Causas comuns:

  • Arquivos grandes sendo injetados no contexto em toda chamada (MEMORY.md crescendo sem controle)
  • System prompt excessivamente longo com redundâncias
  • Cron jobs com frequência alta e contexto desnecessariamente rico
  • Loop de agente consumindo tokens sem produzir resultado
  • Definições de ferramentas extensas enviadas em chamadas que não as usam

Limites de taxa (rate limits) sendo atingidos

Quando o agente bate em rate limits frequentemente, o problema raramente é de capacidade contratada — quase sempre é de padrão de chamadas mal otimizado.

Diagnóstico:

  • Identifique o tipo de limite: RPM (requests) ou TPM (tokens)? Os logs de erro 429 incluem informação sobre qual limite foi atingido.
  • Mapeie os horários de pico: use os logs para encontrar quando as rajadas acontecem.
  • Identifique o agente ou cron responsável: multi-agent pode ter um agente consumindo desproporcionalmente.

Soluções:

  • Implemente backoff exponencial nas chamadas (veja capítulo 7 — seção de quotas)
  • Escalone crons que disparavam no mesmo minuto para horários diferentes
  • Use modelos leves para tarefas de alta frequência
  • Solicite aumento de limite ao provedor se o uso legítimo justificar

Diagnóstico de integração de ferramentas

Falhas em ferramentas (tools) são uma das categorias mais comuns de problemas em agentes operacionais.

Tool calls falhando

Sintoma: o agente tenta usar uma ferramenta mas recebe erro, ou a ferramenta retorna vazio quando deveria retornar dados.

Diagnóstico por tipo de ferramenta:

Ferramenta
Verificação primária
Verificação secundária
Leitura de arquivo
Arquivo existe? Path correto?
Permissão de leitura no sandbox?
Escrita de arquivo
Diretório destino existe?
Sandbox permite escrita nesse path?
Execução de comando
Comando está no PATH?
Escopo elevated necessário? (cap. 10)
Browser/CDP
Processo do browser está rodando?
Perfil correto configurado?
API externa
Credenciais válidas?
Endpoint alcançável da sandbox?

Auditoria de permissões:

# Verifique o escopo da sandbox
openclaw sandbox status
openclaw sandbox scope list

# Se a ferramenta requer acesso não permitido:
# Revise o escopo em config antes de expandir permissões

Resultados de ferramentas não usados corretamente pelo agente

Sintoma: a ferramenta retorna o resultado correto, mas o agente o ignora, o mal-interpreta, ou produz uma resposta que não reflete o que foi retornado.

Diagnóstico de prompt:

  1. Inspecione nos logs o conteúdo exato do tool result que chegou ao modelo.
  2. Verifique se o formato do resultado é o que o agente espera (JSON vs texto plano, estrutura esperada).
  3. Adicione ao system prompt instruções explícitas sobre como processar resultados de ferramentas: "Quando a ferramenta retornar JSON, extraia apenas os campos X e Y para a sua resposta."
  4. Se o resultado é muito longo, o modelo pode estar truncando internamente. Filtre o resultado antes de enviá-lo de volta ao contexto.

Automação de browser instável

Sintoma: o agente usa browser mas falha intermitentemente — às vezes funciona, às vezes não encontra elementos ou trava no carregamento da página.

Causa raiz mais comum: seletores frágeis e timing. Veja o capítulo 11 para estratégias completas. Resumo rápido:

  • Seletor quebrado: o elemento mudou na última atualização do site. Atualize o seletor para um atributo mais estável (data-testid, aria-label) em vez de classes CSS ou posição no DOM.
  • Timing de carregamento: o agente tenta interagir antes do elemento estar renderizado. Adicione espera explícita por visibilidade do elemento, não apenas por carregamento da página.
  • JavaScript assíncrono: conteúdo carregado via SPA não está disponível imediatamente após o page load. Use waitForSelector com timeout adequado.
  • Captcha ou bloqueio: sites detectam automação e bloqueiam. Considere se a automação do site em questão é permitida pelos termos de uso antes de investir em contornar bloqueios.
  • Perfil do browser: confirme que o perfil está configurado corretamente com openclaw browser profiles e que o processo está rodando antes de tentar a automação.

Árvore de decisão para troubleshooting

Fluxo de diagnóstico expandido

Problema percebido
├─ É um problema técnico (gateway, canal, ferramenta)?
│  ├─ Gateway não sobe → status / logs / doctor / port / auth
│  ├─ Canal não entrega → status --probe / token / pairing / policy
│  └─ Ferramenta falha → permissões / sandbox scope / credenciais
│
├─ É um problema de qualidade de resposta?
│  ├─ Resposta genérica → system prompt vago → revise persona e escopo (cap. 18)
│  ├─ Alucinação → adicione grounding / fontes / instrução de incerteza
│  ├─ Ignora instruções → conflito de contexto / reformule instruções
│  └─ Loop ou preso → kill session / adicione condição de parada / circuit breaker
│
├─ É um problema de performance?
│  ├─ Alta latência → modelo muito grande? contexto excessivo? (cap. 7)
│  ├─ Alto consumo de tokens → audit logs / MEMORY.md crescendo? / cron excessivo?
│  └─ Rate limits → backoff / redistribuir horários / modelos menores
│
└─ É um problema de ferramenta/integração?
   ├─ Tool call falha → permissões / sandbox / credenciais
   ├─ Resultado ignorado → prompt debugging / formato do resultado
   └─ Browser instável → seletores / timing / perfil (cap. 11)

Use sempre a mesma metodologia: confirme o sintoma → colete evidências → formule uma hipótese → teste → valide. Nunca mude múltiplas variáveis ao mesmo tempo ou você não saberá o que resolveu o problema.
