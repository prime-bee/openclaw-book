Como usar as referências oficiais sem se perder

Documentação oficial é onde você confirma a verdade operacional do OpenClaw. O problema é que documentação extensa, quando lida sem critério, pode virar ruído: você abre dez páginas, mistura conceitos e sai com menos clareza do que entrou. Este capítulo existe para reduzir esse atrito. Em vez de listar links soltos, ele organiza qual fonte abrir primeiro, quando abrir a segunda e como combinar o livro com a doc oficial.

O papel deste livro não é substituir a documentação viva. Ele serve como estrutura mental, curadoria e trilha didática. A doc oficial entra quando você precisa confirmar sintaxe, entender comportamento detalhado, acompanhar mudanças de versão ou investigar um caso-limite.

Regra prática: use o livro para ganhar contexto e linguagem comum; use a documentação oficial para validar execução, flags, contratos e troubleshooting fino.

Trilhas de consulta por objetivo operacional

Em vez de abrir páginas isoladas, leia em sequência curta. Isso reduz idas e vindas e ajuda a manter contexto.

Objetivo: instalar e operar pela primeira vez

  1. Leia a home da documentação para entender o produto.
  2. Abra a página de gateway para instalação e operação base.
  3. Confirme os comandos na documentação de CLI.
  4. Volte ao livro para receitas de setup e troubleshooting inicial.

Objetivo: automatizar rotinas

  1. Abra cron jobs para entender agendamento exato.
  2. Abra heartbeat para decidir quando usar contexto da sessão principal.
  3. Consulte memória se a automação depende de continuidade.
  4. Volte ao livro para padrões e critérios práticos.

Objetivo: desenhar ambiente seguro

  1. Abra sandboxing para limites de execução.
  2. Revise gateway para bind, auth e operação exposta.
  3. Consulte multi-agent se houver separação de papéis.
  4. Use o capítulo de segurança do livro para transformar isso em política operacional.

Objetivo: depurar comportamento estranho

  1. Comece pelo componente com sintoma mais evidente.
  2. Abra troubleshooting oficial do gateway ou a doc específica do subsistema.
  3. Valide comandos ou flags na CLI.
  4. Cruze com o capítulo 13 do livro para subir e descer a escada de diagnóstico.

Como validar informação sem virar refém de páginas demais

Um erro comum é tentar confirmar tudo em toda parte. Melhor é usar uma sequência curta e repetível:

  1. Encontre o domínio certo. Se a dúvida é de execução, abra CLI ou gateway; se é de conceito, arquitetura ou memória; se é de risco, sandboxing.
  2. Confirme a página canônica. Use a URL oficial mais específica possível, não uma busca ampla pelo site.
  3. Volte ao livro para integrar o que aprendeu. O livro transforma referência em entendimento operacional.
  4. Guarde o padrão. Se a dúvida reaparecer muito, isso é sinal de que o livro merece expansão nesse ponto.

Erros comuns ao consultar referências oficiais

  • abrir a busca cedo demais: primeiro identifique o domínio, depois procure detalhes.
  • misturar páginas conceituais com páginas operacionais: uma explica o porquê, a outra o como.
  • copiar comandos sem contexto: confirme pré-requisitos, ambiente e superfície antes de executar.
  • congelar a doc em memória: documentação viva muda; quando o detalhe importa, reabra a fonte oficial.
