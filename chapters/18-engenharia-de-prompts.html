Engenharia de prompts: da instrução vaga ao agente previsível

Um modelo de linguagem capaz não garante um agente útil. O que separa um agente frustrante de um agente excelente, na maioria dos casos, não é o modelo — é a qualidade das instruções. Engenharia de prompts é a disciplina de escrever, testar e refinar essas instruções para obter comportamento previsível, seguro e de alta qualidade.

Premissa central: o modelo sabe muito mais do que você usa. Engenharia de prompts é a arte de direcionar esse conhecimento para o problema certo, com o formato certo, no momento certo.

1. Anatomia de um system prompt

O system prompt é a instrução permanente que define quem é o agente, o que ele faz e como ele se comporta. É lido em toda interação e define o "envelope" dentro do qual o modelo opera. Um system prompt bem construído tem cinco componentes:

Papel e identidade

Quem é o agente? Para quem trabalha? Qual é o tom — formal, técnico, casual? Sem essa âncora, o modelo fica em modo genérico e produz respostas medianas.

Exemplo: "Você é o assistente técnico da equipe de dados da Empresa X. Seu estilo é direto, técnico e preferência por listas sobre parágrafos longos."

Escopo explícito

O que o agente DEVE fazer e — ainda mais importante — o que NÃO deve. Fronteiras explícitas evitam alucinação de capacidades e ações indevidas.

Exemplo: "Você responde perguntas sobre o sistema de dados interno. Você NÃO dá conselhos jurídicos, NÃO faz compras e NÃO envia comunicados externos sem aprovação."

Restrições operacionais

Regras de negócio, limites de confidencialidade, políticas de resposta. Isso inclui o que fazer quando não sabe, quando a tarefa está fora do escopo, ou quando precisa escalar.

Exemplo: "Se não tiver certeza, diga claramente que não tem certeza antes de responder. Se a tarefa envolver dados de produção, peça confirmação explícita."

Formato de resposta

Extensão esperada, estrutura (lista, parágrafo, tabela, JSON), idioma, nível de detalhe. Sem esse guia, o modelo escolhe formato por conta própria — nem sempre o mais útil para você.

Exemplo: "Respostas curtas: máximo 3 parágrafos. Respostas técnicas: inclua exemplos de código. Sempre responda em português brasileiro."

Exemplos no prompt

Um ou dois exemplos concretos de input/output esperado. Isso reduz ambiguidade mais do que qualquer descrição textual — o modelo aprende mais com exemplos do que com regras.

Exemplo: "Quando perguntado 'como reinicio o gateway?', responda: 'Execute openclaw gateway restart. Para ver se voltou, use openclaw gateway status.' — direto e com código."

Template de system prompt para OpenClaw

# Papel
Você é [nome/função] da equipe [contexto].
Trabalha para [usuário/organização].
Tom: [formal/técnico/casual].

# Escopo
DEVE fazer: [lista explícita de responsabilidades].
NÃO DEVE fazer: [lista explícita de proibições].

# Restrições operacionais
- Se não tiver certeza, diga antes de responder.
- Para ações irreversíveis (deletar, enviar, publicar), peça confirmação.
- Se a tarefa estiver fora do escopo, oriente o usuário a [alternativa].

# Formato de resposta
- Responda em português brasileiro.
- Prefira listas a parágrafos para informações técnicas.
- Inclua exemplos de código quando relevante.
- Respostas de até [N] linhas, exceto quando explicitamente pedido mais detalhe.

# Exemplo de interação ideal
Pergunta: [exemplo de pergunta típica]
Resposta ideal: [exemplo de resposta no formato e nível esperados]

2. Few-shot prompting: ensinar com exemplos

Few-shot prompting é a técnica de incluir exemplos de input/output dentro do próprio prompt para guiar o comportamento do modelo. Em vez de descrever o que você quer (o que é ambíguo), você mostra. Três exemplos bem escolhidos costumam ter mais impacto do que três parágrafos de instruções.

Quando usar few-shot: quando o formato de saída é específico, quando a tarefa tem um padrão que palavras não descrevem bem, ou quando o modelo está produzindo outputs inconsistentes.

Estrutura de few-shot prompting

# Tarefa
Classifique a urgência de mensagens de suporte como: ALTA, MÉDIA ou BAIXA.
Responda apenas com a classificação, nada mais.

# Exemplos
Mensagem: "O sistema de pagamentos está fora e estamos perdendo vendas"
Classificação: ALTA

Mensagem: "Como faço para mudar minha senha?"
Classificação: BAIXA

Mensagem: "O relatório mensal não está sendo gerado desde ontem"
Classificação: MÉDIA

# Agora classifique:
Mensagem: [mensagem do usuário]
Classificação:

Boas práticas para few-shot:

  • Escolha exemplos representativos — cubra casos típicos, não só os extremos.
  • Use 2 a 5 exemplos — menos de 2 pode não ser suficiente; mais de 6 raro traz ganho e ocupa contexto.
  • Mantenha formato consistente — cada exemplo deve ter exatamente a mesma estrutura.
  • Inclua exemplos de borda — mostre o que fazer em casos ambíguos ou difíceis.
  • Atualize exemplos quando o comportamento mudar — exemplos obsoletos confundem o modelo.

3. Chain-of-thought: raciocínio passo a passo

Chain-of-thought (CoT) é a técnica de instruir o modelo a raciocinar explicitamente antes de dar a resposta final. Para tarefas complexas — diagnóstico, planejamento, análise de múltiplas variáveis — pedir que o modelo "pense em voz alta" melhora significativamente a qualidade e a detectabilidade de erros.

Ativando chain-of-thought

# Técnica simples
Antes de responder, analise os seguintes pontos:
1. Qual é o problema central?
2. Que informações tenho e o que está faltando?
3. Quais são as opções possíveis?
4. Qual é a melhor opção e por quê?
Depois, responda com base nessa análise.

# Técnica de scratchpad
<pensamento>
[o modelo raciocina aqui — você pode instruí-lo a usar essas tags]
</pensamento>

Resposta final: [resposta limpa para o usuário]
Cuidado com CoT em produção: cadeia de pensamento aumenta o número de tokens gerados. Em respostas de alto volume ou com restrição de custo, use com moderação ou apenas para tarefas que justificam o custo.

Casos de uso ideais para chain-of-thought no OpenClaw:

  • Diagnóstico de problemas — o agente raciocina sobre sintomas antes de sugerir solução.
  • Planejamento de automações — o agente detalha cada passo antes de executar.
  • Análise de risco — o agente lista prós, contras e riscos antes de recomendar.
  • Revisão de código ou documentos — o agente identifica problemas antes de sugerir correção.

4. Prompts para saídas estruturadas

Sistemas que consomem output de agentes — dashboards, integrações, relatórios automáticos — precisam de saída previsível. Três formatos dominam: JSON, Markdown e tabelas.

Formato
Quando usar
Como solicitar
JSON
Integração com código, APIs, armazenamento estruturado
"Responda apenas com JSON válido, sem texto adicional. Schema: {campo: tipo}"
Markdown
Documentação, relatórios, wikis, saída para humanos
"Formate a resposta em Markdown com cabeçalhos h2, listas e blocos de código quando necessário."
Tabela Markdown
Comparações, dados tabulares, relatórios
"Apresente as informações em formato de tabela Markdown com colunas: [col1, col2, col3]."
Texto simples
Mensagens de chat, respostas conversacionais
"Responda em texto corrido, sem formatação especial."

Solicitando JSON estruturado com schema

Analise a mensagem de suporte e responda APENAS com JSON válido.
Não inclua texto antes ou depois do JSON.

Schema:
{
  "categoria": "string (tecnico|financeiro|administrativo)",
  "urgencia": "string (alta|media|baixa)",
  "resumo": "string (máximo 100 caracteres)",
  "acoes_sugeridas": ["string"]
}

Mensagem: [mensagem do usuário]
Dica de robustez: mesmo com instrução clara, modelos ocasionalmente inserem texto antes ou depois do JSON. Considere usar um parser que busque o bloco JSON dentro da resposta em vez de parsear a string completa.

5. Anti-padrões de prompt

Estes são os erros mais comuns que fazem prompts falharem. Reconhecê-los é mais rápido do que descobri-los por tentativa e erro.

Anti-padrões e como corrigi-los

Anti-padrão
Sintoma
Correção
Instruções vagas
"Seja útil" → respostas genéricas e inconsistentes
Substitua por instruções específicas: "responda em no máximo 3 frases", "inclua código quando relevante"
Contradições internas
"Seja conciso" + "explique em detalhes" → o modelo escolhe um aleatoriamente
Resolva a contradição antes de enviar: "seja conciso em respostas rotineiras; detalhe em questões técnicas"
Scope creep
Prompt acumula regras ao longo do tempo sem curadoria → comportamento imprevisível
Revise e consolide o prompt mensalmente; remova regras que se contradizem ou que não são mais relevantes
Negações sem alternativa
"Não faça X" sem dizer o que fazer → o modelo improvisa
"Não faça X; em vez disso, faça Y"
Instruções implícitas
Você assume que o modelo "sabe" o que você quer → surpresas frequentes
Torne tudo explícito, especialmente o óbvio; o modelo não tem o seu contexto implícito
Prompt monolítico
Um bloco de texto sem estrutura → difícil de manter e de debugar
Use cabeçalhos (# Papel, # Escopo, # Formato) para organizar e facilitar edição

6. System prompts para ferramentas OpenClaw: exemplos reais

Abaixo estão templates de system prompt para casos de uso comuns no OpenClaw. Adapte ao seu contexto — os exemplos são pontos de partida, não fórmulas rígidas.

Agente de operações diárias

# Papel
Você é o assistente operacional de [nome]. Seu trabalho é ajudar
com tarefas técnicas do dia a dia: diagnóstico de sistema, busca de
informações, redação de documentos técnicos e execução de comandos
quando autorizado.

# Escopo
DEVE: responder perguntas técnicas, buscar arquivos no workspace,
redigir rascunhos, diagnosticar problemas comuns, resumir logs.
NÃO DEVE: executar comandos em produção sem confirmação, enviar
comunicações externas, modificar arquivos de configuração críticos.

# Confirmação obrigatória antes de
- Deletar qualquer arquivo
- Reiniciar serviços
- Enviar mensagens para canais externos
- Modificar memória de outros agentes

# Formato
Respostas em português. Prefira bullet points a parágrafos.
Inclua comandos em blocos de código. Seja conciso por padrão;
ofereça detalhes quando perguntado.

Agente de relatórios automáticos (cron)

# Papel
Você é o agente de relatórios diários. Executa de forma autônoma
às [hora] e produz um resumo estruturado para [destinatário].

# Tarefa fixa
1. Leia o log do dia anterior em workspace/logs/
2. Identifique: erros críticos, avisos, operações concluídas
3. Produza um relatório no formato abaixo
4. Salve em workspace/reports/YYYY-MM-DD.md

# Formato do relatório
## Relatório Operacional — [data]
### Resumo executivo (2-3 frases)
### Erros críticos (lista, ou "Nenhum")
### Avisos (lista, ou "Nenhum")
### Operações concluídas (lista)
### Recomendações (se houver)

7. Testes e iteração de prompts

Um prompt nunca está "pronto". Ele é uma hipótese que você testa e refina. Aqui está um processo pragmático:

Etapa
O que fazer
1. Escreva o prompt mínimo
Comece com o essencial: papel, escopo, formato. Não sobrecarregue.
2. Teste com casos típicos
Execute 5-10 interações representativas e avalie: o output foi o esperado?
3. Teste com casos de borda
O que acontece com perguntas fora do escopo? Com inputs ambíguos? Com inglês misturado?
4. Identifique padrões de falha
Onde o agente erra sistematicamente? Há regras que produzem comportamento indesejado?
5. Refine e documente
Faça uma mudança de cada vez. Documente o que mudou e por quê. Teste novamente.
6. Versione
Salve versões do prompt no workspace. Se uma mudança piorar o comportamento, você consegue reverter.
Princípio de teste: se você não consegue prever o output do agente para um dado input, o prompt não está bom o suficiente. Previsibilidade é o objetivo final da engenharia de prompts.

Log de iteração de prompt (template para workspace)

## Prompt: [nome do agente] — histórico de versões

### v1.0 — [data]
Mudança: versão inicial
Resultado: [o que funcionou, o que não funcionou]

### v1.1 — [data]
Mudança: [descrição específica da mudança]
Motivação: [por que foi necessário mudar]
Resultado: [melhoria observada, efeitos colaterais]

### v2.0 — [data]
Mudança: [mudança maior]
Prompt atual: [cole o prompt completo aqui]

Exercícios

Exercício 1 — Auditoria de system prompt: Abra o system prompt do seu agente principal e avalie: ele tem os cinco componentes (papel, escopo, restrições, formato, exemplos)? Qual está faltando? Adicione o que falta e teste com três perguntas típicas. O comportamento mudou?
Exercício 2 — Few-shot para formatação: Escolha uma tarefa que o agente faz com formato inconsistente (resumos, classificações, relatórios). Escreva 3 exemplos de input/output ideal. Adicione ao prompt e compare os resultados antes e depois.
Exercício 3 — Saída estruturada: Peça ao agente para resumir os logs da última semana em formato JSON com campos: data, eventos_criticos, contagem_operacoes, status_geral. Avalie se o JSON foi válido e parseável. Se não foi, refine a instrução.
Exercício 4 — Teste de anti-padrões: Leia o system prompt atual e identifique pelo menos um anti-padrão da lista acima (instrução vaga, contradição, negação sem alternativa). Corrija e documente a mudança no seu log de versões.
Exercício 5 — Iteração documentada: Faça três versões de um prompt para uma tarefa real. Para cada versão, registre o que mudou e o que melhorou. Ao final, você terá um mini-histórico de engenharia de prompts que serve como referência futura.
