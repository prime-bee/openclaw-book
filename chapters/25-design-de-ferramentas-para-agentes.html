Ferramenta boa para agente não é só "uma API exposta ao modelo". É uma interface desenhada para reduzir ambiguidade, limitar risco e transformar intenção em ação observável.

Design de ferramentas para agentes: contratos, ergonomia e blast radius

Quando um agente usa uma ferramenta, ele cruza uma fronteira importante: sai do mundo da linguagem e entra no mundo da ação. A qualidade dessa transição depende do desenho da ferramenta. Se a interface é vaga, o agente improvisa mal. Se a ferramenta aceita tudo, o risco sobe. Se o retorno é confuso, o modelo não sabe o que aconteceu. E se não há observabilidade, ninguém consegue depurar quando algo dá errado.

Tese do capítulo: ferramenta boa é a que torna o caminho certo mais fácil do que o caminho perigoso.

O que uma ferramenta faz de verdade

No ecossistema OpenClaw, ferramentas de primeira classe não existem só para "dar mais poder" ao modelo. Elas existem para estruturar interação com o ambiente: sistema de arquivos, runtime, sessões, memória, browser, cron, web e integrações externas.

Capacidade

A ferramenta expõe algo que o modelo sozinho não consegue fazer: ler, escrever, executar, consultar, agendar, buscar.

Contrato

A interface define entradas, limites, defaults, formatos e erros esperados.

Controle

Políticas de sandbox, allow/deny, perfis de tools e approvals limitam o blast radius.

Feedback

O retorno da ferramenta precisa ser legível o bastante para o modelo e auditável o bastante para humanos.

Sem esses quatro elementos, o agente deixa de operar sobre interfaces bem projetadas e volta a depender de improviso textual.

O primeiro princípio: desenhe para intenção, não para implementação interna

Uma ferramenta ruim costuma vazar detalhes internos demais: flags obscuras, parâmetros acoplados ao backend, formatos difíceis de montar e respostas cheias de ruído operacional. O agente não precisa entender a anatomia completa do seu sistema; ele precisa de uma interface que corresponda à intenção da tarefa.

Ruim
Melhor
dez parâmetros opcionais sem defaults claros
poucos parâmetros obrigatórios, defaults sensatos e exemplos
campo genérico como payload para tudo
campos nomeados por intenção: message, timeoutSeconds, path
retorno cru do backend
resposta resumida com os dados mais acionáveis
erros opacos
erros com causa provável e ação sugerida

Ferramenta bem desenhada é quase uma pequena opinião de produto: ela ensina o uso correto pela própria forma.

Contrato de entrada: o modelo precisa de trilhos

Grande parte do design de ferramentas é design de schema. O agente pensa em linguagem natural; a ferramenta exige estrutura. Quanto mais nítido for esse contrato, menor a chance de o modelo inventar formatos, misturar tipos ou enviar ações perigosas por engano.

Checklist de contrato de entrada

  1. defina o menor conjunto de campos obrigatórios possível
  2. use nomes explícitos e específicos
  3. prefira enums e tipos fechados quando o domínio for pequeno
  4. coloque limites claros: tamanho, range, quantidade, timeout
  5. documente defaults de forma que o agente não precise adivinhar
  6. evite parâmetros mágicos cujo efeito só o mantenedor entende
Exemplo de ferramenta mal desenhada:
{
  "action": "run",
  "mode": "special",
  "data": "..."
}

Exemplo mais saudável:
{
  "action": "run",
  "command": "pytest tests/unit",
  "timeoutSeconds": 120,
  "workdir": "/workspace/app"
}

No segundo caso, a intenção está visível. Isso ajuda tanto o modelo quanto o humano que vai auditar depois.

Contrato de saída: retorne o que ajuda a decidir o próximo passo

Uma ferramenta não termina quando executa a ação. Ela termina quando devolve uma resposta que permite ao agente decidir o que fazer em seguida. Resultado demais vira barulho. Resultado de menos vira cegueira.

Tipo de retorno
O que quase sempre vale incluir
Leitura/consulta
dados pedidos, origem, truncation ou offset quando houver
Escrita
o que mudou, onde mudou, quantos itens foram afetados
Execução
exit code, trecho útil de stdout/stderr, session id quando for assíncrono
Agendamento
identificador do job, schedule efetivo e destino de entrega
Regra útil: toda resposta de ferramenta deveria ajudar o agente a responder uma destas perguntas: "deu certo?", "o que mudou?", "preciso esperar?", "preciso corrigir algo?"

Ferramenta síncrona, assíncrona ou em duas etapas?

Nem toda operação cabe no mesmo envelope. Algumas ações são rápidas e devolvem resultado imediatamente. Outras exigem processo em background, polling eventual e recuperação posterior. Misturar tudo em uma interface só tende a produzir confusão.

Síncrona

Boa para leitura curta, edição pontual, consultas simples e respostas imediatas.

Assíncrona

Boa para jobs demorados, processos de longa duração, execução com log incremental e automações persistentes.

Duas etapas

Boa quando faz sentido separar "disparar" de "acompanhar", como exec + process.

O OpenClaw ilustra bem isso: exec não precisa fingir que tudo acaba no mesmo turno. Ele pode devolver uma sessão e deixar a inspeção incremental para process. Isso reduz timeout artificial e evita respostas gigantes.

Ergonomia para o modelo: menos liberdade caótica, mais liberdade útil

Há uma tentação comum em sistemas agentic: achar que "quanto mais flexível a ferramenta, melhor". Na prática, ferramentas abertas demais empurram decisão demais para o modelo e geram uso inconsistente.

  • Defaults bons reduzem token gasto explicando parâmetros triviais.
  • Campos opcionais poucos e claros reduzem combinações nonsense.
  • Comandos separados por responsabilidade ajudam mais do que uma mega-tool faz-tudo.
  • Nomes previsíveis ajudam o agente a generalizar entre ferramentas semelhantes.

Um bom desenho de ferramentas tem algo de UX conversacional: você está desenhando uma interface para um usuário probabilístico.

Ferramentas pequenas vencem mega-tools na maioria dos casos

Mega-tools parecem elegantes no papel porque centralizam tudo. Mas, para agentes, elas criam contratos enormes, respostas inconsistentes e espaço demais para o modelo escolher o ramo errado. Ferramentas menores, com escopo bem definido, costumam performar melhor.

Padrão
Efeito típico
uma ferramenta para "fazer qualquer request arbitrária"
alto poder, baixa previsibilidade, risco grande
ferramentas específicas por domínio
melhor direcionamento, schemas menores, auditoria mais simples
uma ação genérica com dezenas de submodos
o modelo frequentemente escolhe mal
operações separadas por ciclo de trabalho
intenção mais explícita e troubleshooting melhor

Isso não significa explodir tudo em cinquenta tools minúsculas. Significa achar o ponto em que cada ferramenta representa uma unidade real de trabalho.

Idempotência e reexecução: pense no agente que erra e tenta de novo

Agentes tentam novamente. Às vezes porque a resposta foi truncada; às vezes porque o modelo entendeu que precisava confirmar; às vezes porque uma cadeia de raciocínio levou a repetir o mesmo passo. Se a ferramenta não for pensada para isso, uma repetição inocente pode virar efeito colateral grave.

  • operações de escrita devem idealmente ser idempotentes ou detectar duplicação
  • ações externas sensíveis pedem identificadores de deduplicação ou chaves idempotentes
  • em updates, vale retornar o estado final em vez de "ok" genérico
  • quando reexecutar for perigoso, a ferramenta deve dizer isso claramente
Erro clássico: criar uma tool de envio externo que não distingue "criar", "atualizar" e "reenviar". Para um modelo, isso é pedir duplicidade.

Segurança: o schema também é uma política

Segurança em ferramentas não começa no approval final. Começa no formato da interface. Um schema bem fechado reduz superfície de abuso antes mesmo de a chamada ser avaliada por sandbox, tool policy ou permissão elevada.

Quatro camadas úteis

1. Capacidade

O que a ferramenta consegue fazer em tese.

2. Política

Quais tools e perfis ficam disponíveis para esse agente ou provedor.

3. Sandbox

Em que ambiente a ação roda e com que isolamento.

4. Approval

Quais ações ainda exigem consentimento humano.

Essas camadas não competem; elas se completam. Approval não substitui schema bom. Sandbox não substitui tool policy. E nenhuma delas corrige totalmente uma interface que já convida o modelo a fazer coisa errada.

Ferramentas devem ensinar escopo

Muitas falhas operacionais vêm de ambiguidade de escopo: qual workspace? qual sessão? qual canal? qual destinatário? qual janela de tempo? qual ambiente? Ferramentas maduras pedem esse escopo explicitamente ou o derivam de forma segura.

Pergunta útil ao desenhar uma tool:
"Se o agente usar isso no projeto errado, conta errada ou sessão errada,
como a interface reduz a chance de erro?"

Em memória, sessões e automações isso é especialmente importante. Identificadores claros, labels estáveis e campos de destino evitam que o modelo trate todo contexto como se fosse um único espaço homogêneo.

Observabilidade: se não dá para depurar, a ferramenta ainda está incompleta

Ferramenta boa não só executa; ela deixa rastros úteis. Isso vale para logs, IDs, mensagens de erro, status intermediário e distinção entre sucesso parcial e falha real.

O mínimo observável de uma tool confiável

  1. um identificador rastreável quando a operação for longa ou reconsultável
  2. erros estruturados em vez de texto aleatório
  3. indicação de truncamento ou paginação quando o retorno for parcial
  4. sinais de espera ou continuação quando houver processamento em background
  5. mensagem curta o bastante para o modelo usar sem se perder

Quando a ferramenta só cospe o retorno bruto do backend, o humano até talvez consiga fuçar. O agente, porém, tende a interpretar mal ou desperdiçar tokens lendo ruído.

Como pensar o texto de erro

Erro bom não é o que despeja stack trace inteiro. É o que explica o suficiente para orientar a recuperação.

Erro fraco
Erro útil
invalid request
qual campo falhou, valor recebido e formato esperado
permission denied
se a falha veio de tool policy, sandbox, aprovação ou credencial
timed out
se a operação pode continuar em background e como acompanhar

Isso reduz retry cego, que é uma das coisas que mais degradam agentes em produção.

Exemplo prático: uma tool de edição de arquivo

Pense na diferença entre uma ferramenta genérica que aceita "qualquer comando shell" e uma ferramenta específica para editar texto por substituição exata.

Tool genérica demais

Mais poder, mais risco, mais chance de comando destrutivo e mais dificuldade para auditar intenção.

Tool específica

Menor blast radius, intenção explícita, validação melhor e fluxo mais previsível para mudanças cirúrgicas.

Esse contraste ilustra um princípio importante: quanto mais sensível a ação, mais valiosa costuma ser uma interface estreita.

Quando uma ferramenta deve se recusar a agir

Boa parte da segurança operacional vem da recusa correta. Uma tool madura deve falhar cedo quando:

  • faltam parâmetros essenciais para definir o alvo
  • há ambiguidades fortes que exigem confirmação humana
  • o tamanho da operação ultrapassa o escopo saudável daquela interface
  • a política do ambiente nega a ação
  • a repetição da ação pode causar dano e não há deduplicação

Ferramenta que sempre tenta "ser útil" acaba virando vetor de erro. Às vezes a melhor resposta é um "não" bem explicado.

Receita de desenho para novas tools

  1. Defina a unidade de trabalho: qual tarefa concreta essa tool resolve?
  2. Mapeie o risco: leitura, escrita, rede, execução, persistência, envio externo?
  3. Feche o schema: entradas mínimas, enums, limites, defaults.
  4. Modele a saída: o que o agente precisa saber para seguir?
  5. Pense no retry: o que acontece se o agente repetir a chamada?
  6. Pense na observabilidade: que IDs, logs ou status ajudam na depuração?
  7. Teste com prompts ruins: um modelo mediano consegue usar certo sem muita sorte?

Perguntas que eu faria antes de publicar uma ferramenta

Drill de revisão:
  • Essa interface reduz erro ou só expõe um backend cru ao LLM?
  • O nome dos campos revela intenção humana?
  • Se o agente escolher o alvo errado, a tool ajuda a perceber?
  • Se a chamada repetir, o efeito colateral explode?
  • O retorno mostra claramente sucesso, falha parcial, continuação ou bloqueio?
  • Há um caminho de menor privilégio antes de exigir ação mais poderosa?

Relação com OpenClaw: por que isso importa tanto aqui

OpenClaw se apoia fortemente em ferramentas de primeira classe. Isso é ótimo porque reduz shelling improvisado e dá contratos melhores para o agente. Mas também aumenta a responsabilidade de desenhar essas interfaces com cuidado: a qualidade do comportamento não depende só do modelo, e sim da combinação entre prompt, tooling, política e feedback.

Na prática, muitos "erros do agente" são erros de design da tool: schema frouxo, retorno mal formatado, falta de escopo, permissões confusas ou interface grande demais para o modelo usar com consistência.

Conclusão prática

Ferramentas são parte do produto cognitivo do agente. Não basta que funcionem no backend; elas precisam ser usáveis por um modelo, auditáveis por humanos e seguras por padrão. Quando o contrato é bom, o agente parece mais cuidadoso, mais previsível e mais competente. Quando o contrato é ruim, até um modelo forte parece desatento.

Se você quer melhorar agentes em produção, não olhe só para prompts e modelos. Olhe com carinho brutal para as ferramentas. Quase sempre há muito ganho escondido ali.
